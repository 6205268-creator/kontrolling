# Бизнес-логика

## Доменные сущности

### v2 (текущий backend для UI)

| Сущность | Таблица | Описание |
|----------|---------|----------|
| **Физ. лицо** | `physical_person` | Реальный человек; глобально, не привязан к СНТ. Источник всех платежей. |
| **СНТ** | `snt` | Садовое товарищество (tenant). |
| **Член СНТ** | `snt_member` | Членство: СНТ + физ. лицо за период. Ключ `(snt_id, physical_person_id, date_from)`. Повторное вступление = новая строка. |
| **Участок** | `plot` | Участок в СНТ. `UNIQUE (snt_id, number)`. |
| **Владелец участка** | `plot_owner` | Участок ↔ физ. лицо за период. Один человек — много участков; один участок — разные владельцы во времени. |
| **Счётчик** | `meter` | Счётчик на участке. `meter_type`: `electricity`, `water`. `UNIQUE (snt_id, plot_id, meter_type)`. |
| **Пользователь приложения** | `app_user` | Учётная запись. `role`: `admin` \| `snt_user`; у `snt_user` задан `snt_id`. |

### v1 (core; документы, регистры)

- `owner` («собственник» по СНТ), `plot_owner` (участок ↔ owner), `charge_item`, `doc_accrual` / `doc_accrual_row`, `doc_payment` / `doc_payment_row`, `reg_balance`.

## Бизнес-правила

- **Членство:** вступление → новая строка `snt_member`. Выход → `date_to`. Повторное вступление → новая строка (новое членство).
- **Платежи:** всегда от **физ. лица**. Как **член** → членские взносы; как **владелец участка** → коммунальные (электричество, вода и т.д.) по счётчикам.
- **Счётчики:** один счётчик каждого типа на участок (`snt_id`, `plot_id`, `meter_type` уникальны).
- **Пользователи:**  
  - **admin:** видит все СНТ, может переключать СНТ.  
  - **snt_user:** только свой СНТ (`user.snt_id`). Без переключателя СНТ; нет доступа к данным других СНТ.

**v1:** У документов есть `is_posted`. Только проведённые создают движения в `reg_balance`. Проведение пересчитывает движения; отмена — удаляет их.

---

## Сценарии

### v2

- **Просмотр:** выбор пользователя → (если admin) выбор СНТ → загрузка членов, участков, счётчиков, физ. лиц. Детали физ. лица → модалка с членствами и владением участками.
- **Смена пользователя:** перезагрузка `/me`, затем `/snts` и данных разделов. Для `snt_user` всегда используется его единственный СНТ.

### v1 (core)

- **Проведение:** 1) Начать транзакцию. 2) Удалить старые строки `reg_balance` по документу. 3) Рассчитать новые движения из строк. 4) Вставить в `reg_balance`. 5) `is_posted = TRUE`. 6) Commit.
- **Отмена:** 1) Начать транзакцию. 2) Удалить строки `reg_balance` по документу. 3) `is_posted = FALSE`. 4) Commit. Компенсирующих записей нет.

---

## Расчёты

- **v2:** Расчётов нет. Количества (члены, участки, счётчики) — по длине списков; агрегаты не хранятся.
- **v1:** Движения `reg_balance`: начисления → `amount_debit = amount`, `amount_credit = 0`; оплаты → `amount_debit = 0`, `amount_credit = amount`. Остатки и отчёты — из `reg_balance` (например `GET /reports/balance?on_date=`).
