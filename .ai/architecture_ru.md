# Архитектура

## Компоненты системы

| Компонент | Путь | Назначение |
|-----------|------|------------|
| **Backend v1 (core)** | `backend_snt_core/` | Полный учёт: документы, проводки, `reg_balance`. PostgreSQL. API :8000. |
| **Backend v2** | `backend_snt_v2/` | Справочники и связи: физ. лица, члены, участки, счётчики, пользователи. SQLite. API :8001. |
| **Frontend** | `frontend/` | SPA-подобный UI: выбор пользователя/СНТ, дашборды, таблицы, модалки. Читает только из v2 API. |
| **Статический сервер** | `frontend/run.ps1` | Раздаёт `frontend/` на :3000 (например `python -m http.server`). |

**Вспомогательное:**
- `run-v2.ps1` — инициализация БД (миграции + seed), запуск v2 API, фронтенд-сервера, открытие браузера.
- `PLAN_V2_IMPLEMENTATION.md`, `CHANGELOG_IMPLEMENTATION.md` — план и журнал изменений.

---

## Взаимодействие

- Фронтенд обращается **только** к v2 API (`http://localhost:8001/api`). v1 не используется напрямую.
- **v1** автономен: Swagger/главная, своя БД, запуск через `backend_snt_core/run.ps1`.
- Все эндпоинты v2 (кроме `GET /users`) используют `get_current_user` из `X-User-Id` и SNT-фильтрацию для `snt_user`.

---

## Поток данных

1. **Загрузка страницы:** фронтенд запрашивает `/users`, выбирает пользователя (или из `localStorage`). Дальше отправляет `X-User-Id`.
2. **Смена пользователя:** `/me`, затем `/snts`. admin → показывается переключатель СНТ; `snt_user` → скрыт, фиксирован один СНТ.
3. **Разделы:** запросы `/physical-persons`, `/snt-members?snt_id=`, `/plots?snt_id=`, `/meters?snt_id=`. Для `snt_user` backend игнорирует `snt_id` и использует `user.snt_id`.
4. **Детали физ. лица:** клик по строке → `GET /physical-persons/{id}` → модалка с членствами и владением участками. Для `snt_user` — 404, если у лица нет связи с его СНТ.

**v1 (core):** CRUD по REST → сервисы → БД. Проведение/отмена обновляет `reg_balance`; UI (главная) обращается к v1 API только при использовании.

---

## Внешние интеграции

- **Нет.** Нет сторонних API, очередей, внешних сервисов.
- Фронт подключает Google Fonts (DM Sans) через `<link>`; серверной интеграции нет.

# Архитектура ядра учёта СНТ (v1)

Этот документ описывает 1С‑подобную пересчитываемую модель учёта для СНТ, реализуемую на стеке PostgreSQL + FastAPI + SQLAlchemy.
Файл предназначен для использования в Obsidian (разбиение на секции, ссылки, чёткое выделение сущностей).

---

## 1. Общая концепция

- **Ядро системы** = PostgreSQL (данные) + бизнес‑логика на backend.
- **UI и фронтенд не являются источником истины** и могут быть полностью отключены.
- Все расчёты и инварианты контролируются на стороне ядра:
  - структура БД (FK, CHECK, UNIQUE, NOT NULL);
  - транзакции;
  - сервисы проведения/отмены.

Система реализует 3 слоя данных:

1. **Первичные данные**: справочники, документы, табличные части.
2. **Производные данные**: регистры (движения), пересчитываемые по документам.
3. **Состояние документа**: флаг `is_posted` (проведён / не проведён).

---

## 2. Multi‑tenant модель (СНТ)

- Во **всех таблицах** присутствует поле `snt_id`.
- Отсутствие `snt_id` считается архитектурной ошибкой.
- Основная сущность tenant: таблица `snt`.
- Данные разных СНТ не смешиваются, при необходимости может быть включён Row Level Security.

---

## 3. Структура данных

### 3.1. Справочники (первичные данные)

- `snt`
  - `id` (PK)
  - `name`
  - служебные поля (`created_at`, `updated_at`)

- `plot` (участки)
  - `id` (PK)
  - `snt_id` (FK → `snt`)
  - `number` (номер/обозначение участка)
  - дополнительные атрибуты по необходимости

- `owner` (собственники)
  - `id` (PK)
  - `snt_id` (FK → `snt`)
  - ФИО/ИНН и другие данные

- `plot_owner` (связь участок–собственник)
  - `id` (PK)
  - `snt_id` (FK → `snt`)
  - `plot_id` (FK → `plot`)
  - `owner_id` (FK → `owner`)
  - период действия связи (даты начала/окончания)

- `charge_item` (статьи начислений/услуг)
  - `id` (PK)
  - `snt_id` (FK → `snt`)
  - `name`
  - `type` (членский, целевой, услуга и т.п.)

Все справочники:

- имеют `snt_id NOT NULL`;
- индексируются по `snt_id`;
- связаны между собой через `FOREIGN KEY`.

---

## 4. Документы и табличные части

### 4.1. Общие правила

- Документ:
  - имеет уникальный идентификатор `id`;
  - принадлежит ровно одному СНТ (`snt_id`);
  - имеет флаг состояния `is_posted`;
  - хранит только вводимые пользователем данные (без производных расчётов).

- Табличные части документов:
  - являются первичными данными;
  - всегда ссылаются на документ и СНТ;
  - не содержат производных итогов.

### 4.2. Документы первого инкремента

**Документ начисления** (`doc_accrual`):

- Шапка:
  - `id`, `snt_id` (FK → `snt`)
  - `number`
  - `date`
  - `is_posted`

- Табличная часть `doc_accrual_row`:
  - `id`
  - `snt_id` (FK → `snt`)
  - `doc_id` (FK → `doc_accrual`)
  - `plot_id` (FK → `plot`)
  - `charge_item_id` (FK → `charge_item`)
  - `amount` (сумма начисления)
  - при необходимости: период действия (`period_from`, `period_to`)

**Документ оплаты** (`doc_payment`):

- Шапка:
  - `id`, `snt_id`
  - `number`
  - `date`
  - `is_posted`

- Табличная часть `doc_payment_row`:
  - `id`
  - `snt_id`
  - `doc_id` (FK → `doc_payment`)
  - `plot_id` и/или `owner_id`
  - `charge_item_id`
  - `amount` (сумма оплаты)

Во всех документах и строках соблюдается инвариант:

- `snt_id` строк = `snt_id` документа;
- проверка реализуется через FK/ограничения/триггер (в БД, а не в коде).

---

## 5. Регистры (производные данные)

### 5.1. Общие правила

- Регистр содержит **только производные данные**.
- Каждая запись регистра:
  - всегда ссылается на документ;
  - всегда ссылается на строку документа;
  - не существует без документа и строки;
  - всегда пересчитывается заново при проведении.

### 5.2. Регистр взаиморасчётов (`reg_balance`)

Назначение: хранить обороты по начислениям и оплатам в разрезе участков/собственников/статей.

Основные поля:

- `id` (PK)
- `snt_id` (FK → `snt`)
- `document_id` (FK → соответствующий документ)
- `document_row_id` (FK → строка документа)
- `document_type` (enum: например, `ACCRUAL`, `PAYMENT`)
- `date` (дата движения, как правило дата документа)
- `plot_id` (FK → `plot`)
- `owner_id` (опционально, FK → `owner`)
- `charge_item_id` (FK → `charge_item`)
- `amount_debit`
- `amount_credit`

Правила заполнения:

- Начисления (`doc_accrual`):
  - `amount_debit = amount`
  - `amount_credit = 0`

- Оплаты (`doc_payment`):
  - `amount_debit = 0`
  - `amount_credit = amount`

---

## 6. Алгоритм проведения и отмены

### 6.1. Проведение документа

Операция проведения выполняется **только** на стороне backend/БД и реализуется строго по шагам:

1. Начать транзакцию.
2. Удалить все старые движения по `document_id` (и `document_type`) из всех затрагиваемых регистров.
3. На основе текущих данных документа и его табличных частей рассчитать новые движения.
4. Вставить новые движения в регистры.
5. Установить документу флаг `is_posted = TRUE`.
6. Зафиксировать транзакцию.

Если любой шаг завершается с ошибкой, вся операция откатывается.

### 6.2. Отмена проведения

Отмена проводится только так:

1. Начать транзакцию.
2. Удалить все движения по `document_id` (и `document_type`) из регистров.
3. Установить `is_posted = FALSE` у документа.
4. Зафиксировать транзакцию.

Данные документа не удаляются, компенсирующие записи не создаются.

---

## 7. Backend‑слой (FastAPI + SQLAlchemy)

### 7.1. Роли слоёв

- **Модели БД (SQLAlchemy)**:
  - соответствуют таблицам PostgreSQL;
  - описывают только структуру, без бизнес‑логики.

- **Сервисы доменной логики**:
  - реализуют работу с документами (CRUD);
  - реализуют проведение, отмену проведения и перепроведение;
  - строго следуют описанным выше алгоритмам.

- **REST‑API (FastAPI роутеры)**:
  - принимают команды в формате JSON;
  - не содержат бизнес‑логики;
  - вызывают сервисы и возвращают результаты.

Фронтенд не проводит документы, не пересчитывает регистры и не принимает архитектурных решений.

---

## 8. Проверяемость сценариев

Любой бизнес‑сценарий должен:

- быть выполнимым только через SQL и backend (без UI);
- быть понятным через схему БД и запросы к таблицам/регистрам;
- допускать ручную проверку и вмешательство через pgAdmin/DBeaver/SQL‑консоль.

Если сценарий нельзя объяснить с точки зрения SQL‑запросов к таблицам и регистрам, архитектура считается нарушенной.

# Архитектура СНТ v2: физ. лицо и член СНТ разделены

Член СНТ (membership) и физ. лицо (physical person) — **разные сущности**.  
Ключ для «членства»: **SNT id + Physical person id** (+ период).  
Один физ. лицо может иметь несколько членств (выход и повторное вступление = новое членство), несколько участков, несколько счётчиков, несколько оплат.

---

## 1. Ключевые сущности (кратко)

| Таблица (English) | По‑русски | Описание |
|-------------------|-----------|----------|
| `physical_person` | физ. лицо | Реальный человек, независим от СНТ. Источник денег: он платит. |
| `snt` | СНТ, товарищество | Садовое некоммерческое товарищество (аналог — как сейчас). |
| `snt_member` | член СНТ, членство | Факт членства физ. лица в СНТ за период. Выход и возврат = новое членство. |
| `plot` | участок | Участок в СНТ (аналог — как сейчас). |
| `plot_owner` | владелец участка | Связь участок ↔ физ. лицо за период (кто владеет, когда). |
| `meter` | счётчик, прибор учёта | Электричество, вода и т.п. Привязан к участку. |
| `charge_item` | статья начислений, услуга | Членский взнос, целевой, электроэнергия, вода и т.д. |

**Кто за что платит (источник денег = физ. лицо):**

- Как **член СНТ** (через `snt_member`) → обязан платить **членские взносы** (member fees).
- Как **владелец участка** (через `plot_owner`) → платит за **электричество** (по счётчику), за **воду** (по счётчику) и другие услуги по потреблению.

---

## 2. Схема БД (таблицы, ключи, связи)

### 2.1. `physical_person` (физ. лицо)

Человек существует **глобально**, не привязан к конкретному СНТ.

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `full_name` | VARCHAR(250) NOT NULL | ФИО |
| `inn` | VARCHAR(12) NULL | ИНН (аналог — налог. учёт) |
| `phone` | VARCHAR(50) NULL | Телефон |
| `created_at` | TIMESTAMPTZ | Создано |
| `updated_at` | TIMESTAMPTZ | Обновлено |

**Уникальность:** `id` — PK. При необходимости — `inn` UNIQUE (если ведём учёт по ИНН).

**Связи:** к `physical_person` ссылаются `snt_member`, `plot_owner`, документы оплат (плательщик = физ. лицо).

---

### 2.2. `snt` (СНТ, товарищество)

Без изменений по смыслу (аналог — текущая таблица СНТ).

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `name` | VARCHAR(200) NOT NULL UNIQUE | Название СНТ |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

---

### 2.3. `snt_member` (член СНТ, членство)

Связка **СНТ + физ. лицо** за период. Вступление → новая строка. Выход → `date_to`. Повторное вступление → **новая** строка (новое членство, новая «история»).

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `snt_id` | FK → `snt.id` | СНТ |
| `physical_person_id` | FK → `physical_person.id` | Физ. лицо |
| `date_from` | DATE NOT NULL | Дата вступления (аналог — начало членства) |
| `date_to` | DATE NULL | Дата выхода; NULL = по настоящее время |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

**Уникальность:**

- `UNIQUE (snt_id, physical_person_id, date_from)` — одно членство на одну пару (СНТ, физ. лицо) и одну дату вступления.
- Один физ. лицо может иметь **несколько** строк в `snt_member` по одному СНТ (разные периоды: вышел — снова вступил).

**Ключевая пара для «членства»:** `(snt_id, physical_person_id)` + период `(date_from, date_to)`.

**Связи:**  
Членские взносы привязаны к **членству** (т.е. к `snt_member`): как только физ. лицо стало членом СНТ, у него появляются обязательства по членским взносам.

---

### 2.4. `plot` (участок)

Участок принадлежит одному СНТ (аналог — текущая модель участка).

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `snt_id` | FK → `snt.id` | СНТ |
| `number` | VARCHAR(50) NOT NULL | Номер участка |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

**Уникальность:** `UNIQUE (snt_id, number)` — в рамках одного СНТ номер участка не повторяется.

---

### 2.5. `plot_owner` (владелец участка)

Связь **участок ↔ физ. лицо** за период. Один участок может в разное время принадлежать разным физ. лицам; одно физ. лицо может владеть несколькими участками.

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `snt_id` | FK → `snt.id` | СНТ (для составных FK, наследуется от участка) |
| `plot_id` | FK → `plot.id` | Участок |
| `physical_person_id` | FK → `physical_person.id` | Владелец (физ. лицо) |
| `date_from` | DATE NOT NULL | С какого числа владеет |
| `date_to` | DATE NULL | По какое; NULL = по н.в. |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

**Уникальность:** `UNIQUE (snt_id, plot_id, physical_person_id, date_from)` — одна запись на комбинацию (СНТ, участок, физ. лицо, дата начала владения).

**Связи:**

- Участок и владелец всегда в рамках одного СНТ (через `snt_id` у `plot` и в `plot_owner`).
- Оплаты за электричество, воду и т.п. привязываем к **владельцу участка** (т.е. к `plot_owner` / физ. лицу как владельцу).

---

### 2.6. `meter` (счётчик, прибор учёта)

Прибор учёта привязан к участку. Электричество, вода и т.д. — разные типы счётчиков.

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `snt_id` | FK → `snt.id` | СНТ |
| `plot_id` | FK → `plot.id` | Участок |
| `meter_type` | VARCHAR(50) NOT NULL | Тип: `electricity` (электроэнергия), `water` (вода) и т.п. |
| `serial_number` | VARCHAR(100) NULL | Заводской номер |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

**Уникальность:** `UNIQUE (snt_id, plot_id, meter_type)` — на один участок один счётчик данного типа (например, одна вода, одно электричество).

**Связи:**  
Начисления за электроэнергию/воду идут по участку и типу счётчика; платит **владелец участка** (физ. лицо через `plot_owner`).

---

### 2.7. `charge_item` (статья начислений, услуга)

Без изменений по смыслу (аналог — текущий справочник статей).

| Колонка | Тип | Описание |
|--------|-----|----------|
| `id` | PK | Уникальный идентификатор |
| `snt_id` | FK → `snt.id` | СНТ |
| `name` | VARCHAR(250) NOT NULL | Название (напр. «Членские взносы», «Электроэнергия ОДН») |
| `type` | VARCHAR(50) NOT NULL | Тип: `member_fee` (членский), `target` (целевой), `service` (услуга) и т.д. |
| `created_at` | TIMESTAMPTZ | |
| `updated_at` | TIMESTAMPTZ | |

**Уникальность:** `UNIQUE (snt_id, name)` — в рамках СНТ название статьи не повторяется.

---

## 3. Связи (диаграмма)

```
                    ┌─────────────────────┐
                    │  physical_person    │
                    │  (физ. лицо)        │
                    │  id, full_name, …   │
                    └──────────┬──────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         │                     │                     │
         ▼                     ▼                     ▼
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│   snt_member    │  │   plot_owner    │  │  doc_payment    │
│   (член СНТ)    │  │ (владелец       │  │  (оплата)       │
│ snt_id +        │  │  участка)       │  │  payer =        │
│ physical_       │  │ plot_id +       │  │  physical_      │
│ person_id       │  │ physical_       │  │  person_id      │
│ date_from/to    │  │ person_id       │  └─────────────────┘
└────────┬────────┘  └────────┬────────┘
         │                    │
         │                    │
         ▼                    ▼
┌─────────────────┐  ┌─────────────────┐
│      snt        │  │      plot       │
│ (СНТ)           │  │ (участок)       │
└─────────────────┘  └────────┬────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │     meter       │
                    │ (счётчик:       │
                    │  эл‑во, вода)   │
                    └─────────────────┘
```

- **Член СНТ:** ключ — пара `(snt_id, physical_person_id)` + период. Один физ. лицо → несколько членств (разные периоды).
- **Владелец участка:** участок ↔ физ. лицо через `plot_owner`. Один физ. лицо → несколько участков.
- **Плательщик:** всегда физ. лицо (как член — взносы; как владелец — услуги по участку/счётчикам).

## 4. Уникальность записей (сводка)

| Таблица | Ограничение уникальности |
|--------|---------------------------|
| `physical_person` | `id` PK |
| `snt` | `id` PK, `name` UNIQUE |
| `snt_member` | `UNIQUE (snt_id, physical_person_id, date_from)`; при повторном вступлении — новая строка |
| `plot` | `UNIQUE (snt_id, number)` |
| `plot_owner` | `UNIQUE (snt_id, plot_id, physical_person_id, date_from)` |
| `meter` | `UNIQUE (snt_id, plot_id, meter_type)` |
| `charge_item` | `UNIQUE (snt_id, name)` |

## 5. Документы начислений и оплат (кратко)

- **Начисления (accrual):**
  - По **членским взносам** — привязка к `snt_member` (член СНТ).
  - По **электричеству, воде** и др. услугам — к `plot` + `meter`; ответственный — владелец участка (`plot_owner` → `physical_person`).

- **Оплаты (payment):**
  - Плательщик всегда **физ. лицо** (`physical_person_id`).
  - Можно хранить ссылку на `snt_member` (если платят как член) и/или на `plot_owner` (если платят как владелец), в зависимости от правил учёта.

Детализацию табличных частей документов (строки начислений/оплат) и регистров можно вынести в отдельный раздел, сохраняя multi-tenant через `snt_id` во всех таблицах.

## 6. Отличия от текущей схемы (v1)

| Было (v1) | Стало (v2) |
|-----------|------------|
| `owner` (собственник) привязан к СНТ, по сути «член + владелец» в одном | Разделены **физ. лицо** (глобально) и **член СНТ** (snt_member). Владение участком — отдельно (`plot_owner`) |
| `plot_owner` связывает участок и `owner` | `plot_owner` связывает участок и **физ. лицо** |
| Нет явного «членства» с историей (выход/возврат) | Членство = `snt_member` с `date_from`/`date_to`; повторное вступление = новая запись |
| Нет сущности «счётчик» | Появилась `meter` (счётчик) на участок, тип — электричество, вода и т.д. |
| Плательщик не формализован отдельно | Источник денег — **физ. лицо**; платит как член (взносы) и как владелец (услуги по участку) |
