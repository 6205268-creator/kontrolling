**Что проанализировано:**

- Проект находится в фазе чистого проектирования: 0 строк кода приложения
- Выявлен «синдром архитектурного астронавта» — overengineering для задачи учёта СТ

- 35 фич = реалистично 6-12 месяцев до первого рабочего продукта
- Критическая зависимость от AI-ассистента при отсутствии глубоких знаний стека

### 2. Принятие архитектурного решения

**Решение:** Переход с `Python/FastAPI + PostgreSQL` на `1С:Enterprise + OData + Vue 3`.

**Обоснование:**

| Критерий | Старый стек | Новый стек |
|----------|-------------|------------|
| Backend | Писать с нуля (19 фич) | Уже есть (1С) |
| БД | PostgreSQL (настраивать, бэкапить) | Встроена в 1С |
| Бизнес-логика | Python (незнакомый язык) | 1С (родная среда) |
| Мультифирменность | Писать самому | Разделитель данных 1С |
| Время до MVP | 6-12 месяцев | ~2 месяца |
| Экспертиза | Поверхностная | Глубокая |

## Архитектурные решения сессии

| Решение | Обоснование |
|---------|-------------|
| 1С как единственный backend | Бизнес-логика уже там; разработчик знает платформу |
| OData вместо собственного REST | Встроен в 1С, не требует лицензий, покрывает CRUD |
| Отказ от промежуточного backend'а | KISS; OData достаточно для нужд фронтенда |
| Vue 3 + PrimeVue | Современный стек, Cursor хорошо с ним работает; PrimeVue — готовые компоненты |
| Nginx как единственный proxy | Минимальная инфраструктура; проксирует /odata/ на 1С |
| 1С в Git как XML | Каждый объект = отдельный файл; читаемая история изменений через EDT |
