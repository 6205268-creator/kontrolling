# Workflow Orchestration & Task Management

Источник правды по планированию, подагентам, верификации и управлению задачами. Агент должен следовать этому документу при любой нетривиальной работе.

---

## 1. Workflow Orchestration

### 1.1 Plan Node Default
- Входить в режим плана для **любой** нетривиальной задачи (3+ шага или архитектурные решения).
- Если что-то пошло не так — **ОСТАНОВИТЬСЯ** и перепланировать сразу, не продолжать «давить».
- Использовать режим плана и для шагов верификации, не только для реализации.
- Писать детальные спецификации заранее, чтобы снизить неоднозначность.

### 1.2 Subagent Strategy
- Использовать подагентов свободно, чтобы держать основной контекст чистым.
- Передавать исследование, разведку и параллельный анализ подагентам.
- Для сложных проблем — подключать больше «вычислительной мощности» через подагентов.
- Один подагент — одна чёткая линия работы (focused execution).

### 1.3 Self-Improvement Loop
- После **любой** правки по замечанию пользователя: обновлять `tasks/lessons.md` паттерном/уроком.
- Писать правила для себя, которые предотвращают ту же ошибку.
- Итеративно улучшать эти уроки, пока частота ошибок не снизится.
- В начале сессии просматривать уроки для релевантного контекста проекта.

### 1.4 Verification Before Done
- Никогда не помечать задачу выполненной, не доказав, что она работает.
- При релевантности — сравнивать поведение main и своих изменений (diff behavior).
- Спрашивать себя: «Утвердил бы это staff engineer?»
- Запускать тесты, проверять логи, демонстрировать корректность.

### 1.5 Demand Elegance (Balanced)
- Для нетривиальных изменений: остановиться и спросить «есть ли более элегантный способ?»
- Если фикс ощущается как костыль: «Зная всё, что знаю сейчас — реализовать элегантное решение».
- Для простых, очевидных фиксов этот шаг пропускать — не переинженерить.
- Критически оценивать свою работу перед выдачей результата.

### 1.6 Autonomous Bug Fixing
- При отчёте о баге — **просто исправить**. Не просить подсказок.
- Опираться на логи, ошибки, падающие тесты — затем устранять причину.
- Не переключать контекст пользователя: всё делать самому.
- Исправлять падающие CI-тесты без дополнительных инструкций.

## 2. Task Management

1. **Plan First**: писать план в `tasks/todo.md` с проверяемыми пунктами (checkable items).
2. **Verify Plan**: отчитаться перед началом реализации (check-in).
3. **Track Progress**: отмечать пункты выполненными по ходу работы.
4. **Explain Changes**: на каждом шаге давать краткое резюме изменений (high-level summary).
5. **Document Results**: добавлять секцию review в `tasks/todo.md` по завершении.
6. **Capture Lessons**: после исправлений по замечаниям обновлять `tasks/lessons.md`.

---

## 3. Core Principles

- **Simplicity First**: каждое изменение делать максимально простым. Минимальный объём затронутого кода.
- **No Laziness**: искать корневые причины. Никаких временных фиксов. Уровень senior developer.
- **Minimal Impact**: изменения только в необходимом. Не допускать появления новых багов.
