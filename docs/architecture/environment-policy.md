# Политика окружений (Environment Policy)

**Версия:** 1.0  
**Дата:** 2026-03-01  
**Владелец:** DevOps / Архитектор

Документ описывает роли окружений, источники конфигурации, обязательные переменные и правила работы с секретами для проекта Controlling.

---

## 1. Окружения

| Окружение      | Назначение | Источник переменных | Где лежит .env / секреты |
|----------------|------------|----------------------|---------------------------|
| **local dev**  | Разработка на машине разработчика | Файл `.env` в каталоге backend | `backend/.env` (не коммитить; есть `backend/.env.example`) |
| **CI**         | Автоматические тесты при push/PR  | Переменные не требуются для unit/API тестов; при необходимости — GitHub Actions secrets | Секреты в настройках репозитория; тесты используют in-memory SQLite |
| **production** | Боевой запуск через Docker         | Переменные заданы в `docker-compose.yml` (или в .env для compose) | В production — секреты через переменные окружения хоста или vault; не хранить в репозитории |

**Staging:** на текущий момент отдельное окружение staging не используется. При появлении — добавить строку в данную таблицу и описать источник переменных и владение секретами.

---

## 2. Backend: загрузка конфигурации и рабочий каталог

- Конфигурация приложения загружается **только** через слой конфигурации: `backend/app/config.py` (pydantic-settings).
- **Источник:** файл `.env` и/или переменные окружения. Приоритет: переменные окружения переопределяют значения из `.env`.
- **Путь к .env:** pydantic-settings ищет файл `env_file=".env"` **относительно текущей рабочей директории (CWD)**. Поэтому:
  - **При локальном запуске** (uvicorn, pytest, скрипты) **рабочим каталогом должна быть `backend/`**. Тогда подхватывается `backend/.env`.
  - Альтернатива (если запуск из корня репозитория): явно задать переменные окружения или изменить код конфига так, чтобы путь к `.env` был относительно корня репозитория (требует согласования с Backend и обновления документации).
- **Рекомендация:** все команды backend выполнять из каталога `backend/` (см. AGENTS.md, раздел Build & Development).

---

## 3. Обязательные переменные (backend)

| Переменная | local dev | CI | production |
|------------|-----------|-----|------------|
| `DATABASE_URL` | Да (PostgreSQL или по желанию SQLite) | Нет (тесты выставляют SQLite в conftest) | Да (PostgreSQL) |
| `SECRET_KEY`   | Да (можно значение из .env.example для dev) | Не используется тестами | Да (уникальный секрет) |
| `ALGORITHM`    | Опционально (по умолчанию HS256) | — | Опционально |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Опционально | — | Опционально |
| `PROJECT_NAME` | Опционально | — | Опционально |

Для local dev полный перечень и пример значений — в `backend/.env.example`.

---

## 4. Секреты

- **Не хранить** секреты (пароли БД, `SECRET_KEY`, токены) в репозитории. Файл `backend/.env` в `.gitignore`.
- **Владелец секретов:** локально — разработчик; CI — настройки GitHub (secrets); production — DevOps/ответственный за деплой (переменные окружения хоста или секретное хранилище).
- При добавлении новых окружений (например staging) в документе указывать, кто отвечает за секреты и где они хранятся.
