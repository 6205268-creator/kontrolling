# Архитектурные решения (ADR)

Индекс принятых Architecture Decision Records и процесс их ведения.

## Процесс ведения ADR

### Когда создавать ADR

- Принято **значимое** архитектурное решение: выбор технологий, структура модулей, интеграции, политики безопасности, форматы данных на границах системы.
- Изменение существующего решения (новая версия ADR или superseded-by).

Не требуют отдельного ADR: правки опечаток в документации, рефакторинг без изменения контрактов, обновление версий библиотек в рамках текущего стека.

### Формат ADR

Каждый ADR — отдельный файл в этой папке. Рекомендуемая структура:

1. **Контекст (Context)** — проблема, ограничения, почему решение необходимо.
2. **Решение (Decision)** — что принято, в чём суть.
3. **Последствия (Consequences)** — плюсы, минусы, что нужно учитывать дальше.

Дополнительно: статус (Принято / Отклонено / Superseded), дата, ссылки на связанные документы.

### Кто инициирует и кто утверждает

- **Инициирует:** любой участник команды (разработчик, архитектор, техлид) при необходимости зафиксировать решение.
- **Утверждает:** Lead Architect или Tech Lead (согласование с контекстом проекта и AGENTS.md). До утверждения ADR в индексе помечается как «Предложено» или «Черновик».

После утверждения ADR добавляется в индекс ниже; изменения существующего решения — через новый ADR со ссылкой «Supersedes ADR-XXXX».

### Именование файлов

- Формат: `NNNN-short-title.md` (например `0001-canonical-tech-stack.md`).
- Номера — сквозные, с ведущими нулями (0001, 0002, …).
