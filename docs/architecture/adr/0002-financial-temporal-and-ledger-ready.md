# ADR 0002: Временная модель финансов и подготовка к ledger (ledger-ready)

**Статус:** Предложено (черновик)  
**Дата:** 2026-03-04  
**Контекст:** Требуется зафиксировать архитектурную позицию по необратимости операций, балансу на дату и пути к юридически корректной модели без внедрения полного ledger/event sourcing.

---

## Контекст

- Проект — MVP для учёта СТ: без двойной записи, без плана счетов, без event sourcing.
- Необходимо: необратимость финансовых операций, воспроизводимая история, путь к юридически корректной модели.
- Выбран принцип: **Append-only**, исправления через новые операции, отмена необратима, баланс — производная от событий.
- Цель: архитектура **ledger-ready**, но не **ledger-heavy**.

---

## Решение

### 1. Временная модель (две оси)

- **Event Time** — экономическая дата операции (уже есть: `accrual_date` у Accrual, `payment_date` у Payment). Семантически это и есть event_date; отдельный столбец `event_date` не вводим, роль полей фиксируем в глоссарии и в правиле баланса.
- **System Time** — когда операция создана/отменена: `created_at`, `cancelled_at` (последние два — добавить).

### 2. Обязательные поля во всех финансовых операциях (Accrual, Payment, при необходимости Expense)

Добавить:

- `created_at` — уже есть; зафиксировать как NOT NULL и единый источник system time создания.
- `cancelled_at` (datetime, nullable) — момент отмены.
- `cancelled_by_user_id` (UUID, nullable) — кто отменил.
- `cancellation_reason` (text, nullable) — причина (аудит/защита).
- `operation_number` (unique, человекочитаемый) — для документов и аудита.

Поле экономической даты уже есть: у Accrual — `accrual_date`, у Payment — `payment_date`. В правилах баланса и в коде использовать их как Event Time.

### 3. Правило участия операции в балансе на дату X

Операция участвует в балансе на дату **X**, если и только если:

- **Event date** ≤ X (для Accrual: `accrual_date`, для Payment: `payment_date`),
- **created_at** ≤ X (если в будущем появятся backdated-операции),
- и **(status ≠ cancelled)** ИЛИ **(cancelled_at > X)** (т.е. на дату X операция ещё не была отменена).

Формула баланса: сумма участвующих начислений (с учётом статуса applied) минус сумма участвующих платежей (confirmed), с применением этого временного правила.

### 4. Отмена только через доменную операцию

- Логику отмены переносить в сущность (Rich Domain): например `Payment.cancel(cancelled_by, reason, now)`.
- Use case вызывает `entity.cancel(...)` и сохраняет через repository.
- Запрещено: восстановление статуса после отмены, изменение `amount` после создания, UPDATE суммы через ORM в обход домена.

### 5. Защита суммы (immutability)

- `amount` не изменять после создания операции.
- В домене — без setter; при необходимости использовать value object Money и/или защиту на уровне ORM/валидации при update.

### 6. BalanceCalculator как доменный сервис

- Расчёт баланса — только через отдельный сервис (Domain Service), с контрактом вида:
  `calculate(financial_subject_id, as_of_date: date | None) -> Money`.
- Не считать баланс в контроллере, не в ORM-модели, не разбросанным SQL.

### 7. Domain Events (минимальная заготовка)

- При подтверждении/отмене платежа и начисления генерировать доменные события (PaymentConfirmed, PaymentCancelled, AccrualApplied, AccrualCancelled).
- Пока можно не персистить их в БД (логировать/держать в памяти), но контракт событий зафиксировать для будущего ledger/аудита.

### 8. Будущий Ledger (только проектирование)

- Спроектировать, но не внедрять сущность FinancialEntry (id, financial_subject_id, debit, credit, event_date, created_at, source_operation_id).
- Зафиксировать в ADR: текущая модель остаётся агрегатной; переход к ledger — через Domain Events и проекции, без переписывания домена.

---

## Последствия

- **Плюсы:** Временная корректность баланса, необратимость операций, аудит отмен, путь к ledger без слома текущей модели.
- **Минусы:** Несколько новых полей и миграций; отмена только через домен и репозиторий.
- **Обязательно:** Обновить схему данных и диаграммы до кода (Schema First); добавить тесты на правило участия в балансе на дату и на запрет изменения amount.
