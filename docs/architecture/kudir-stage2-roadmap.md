# КУДИР: этап 2 — расширение книги (лёгкий роадмап)

Назначение: не забыть направление развития после разделов 1–2 КУДИР.

Статус: ориентир для проектирования и кода. Детали реализации и продуктовые ограничения, включая тарифы и вопрос бесплатного доступа, в этом файле намеренно не фиксируются — решаются позже.


## 1. Цель этапа

- Книга КУДИР собирается из секций через единый контракт провайдера секций и DTO, без одного большого SQL на всю книгу.
- Добавить возможность включать разделы 3–7, когда появится учёт по наёмным работникам: зарплата, НДФЛ, ФСЗН, пособия — по нормативной структуре книги для СТ.
- Не ломать разделы 1–2: источники данных (начисления, платежи, расходы) остаются как сейчас.


## 2. Два контура учёта

Первый контур — взносы и платежи членов. Это уже принятая линия: начисления, виды взносов, оплаты. В КУДИР относится к разделу 1 и при необходимости к смежной логике.

Второй контур — наём и всё, что от зарплаты: сотрудники, начисления и выплаты, удержания, взносы и выплаты по линии ФСЗН. В КУДИР относится к разделам 3–7.

В интерфейсе оба контура могут быть рядом, например в разделе Финансы, но данные и правила не смешиваются: взносы членов и зарплата — разные вещи.


## 3. Где живёт наём в системе

Слово payroll здесь и в переписке по проекту — не термин из закона, а короткое рабочее имя на английском. По-русски это то же самое, что: зарплатный учёт, учёт по наёмным работникам, расчёт и выплата зарплаты. По содержанию сюда входят зарплата сотрудников, удержанный НДФЛ, страховые взносы и перечисления по линии ФСЗН, пособия — всё, что попадает в разделы КУДИР 3–7 и не является взносами членов СТ.

- Выделить отдельную зону ответственности. В коде и документах её часто называют payroll (зарплатный учёт, наём). Всё, что нужно для строк КУДИР 3–7 и для учёта наёма, не сваливать в модули взносов и начислений членов.
- Конкретный путь в репозитории (отдельный модуль payroll (зарплатный блок) или вложенность в существующий модуль) здесь не фиксируется — выбирается при первой реализации, с опорой на принятые в проекте границы модулей.
- Принцип YAGNI: не создавать заранее пустые слои domain, application, infrastructure, api. Когда появятся сущности и API, структуру выровнять под чистую архитектуру, как в остальных модулях.


## 4. Связь с reporting и КУДИР

- Сборка отчёта остаётся в модуле reporting: сценарий уровня «собрать КУДИР» вызывает набор зарегистрированных провайдеров секций.
- Провайдеры секций 3–7 при реализации читают данные из зоны наёма через порты и интерфейсы или через прикладной слой наёмного модуля, а не дублируют бизнес-правила внутри SQL отчёта.
- Если наёмный учёт ещё не сделан, провайдеры 3–7 не регистрируются или отдают пустую секцию — точное поведение согласовать при реализации. Главное: не ломать разделы 1–2.


## 5. Безопасность

- Как и для остальных отчётов: доступ по ролям, проверка принадлежности к cooperative_id, без доступа к чужому СТ.
- Если появятся загрузка и выгрузка файлов: проверка формата, ограничение размера, права только у доверенных ролей. Персональные данные — по политике проекта, уточнять при вводе функций.


## 6. Поток данных (словами)

Пользователь СТ (председатель или казначей) запрашивает сборку КУДИР.

Модуль reporting собирает книгу: тянет данные из контура взносов, платежей и расходов и, когда будет готово, из зоны наёма payroll (зарплата наёмных, налоги и взносы с неё, ФСЗН).

Итог: один отчёт КУДИР из разных источников, без смешивания правил взносов и зарплаты внутри одного механизма.


## 7. Чеклист на потом

1. Определить минимальную модель данных наёма: сотрудник, период, суммы по статьям, привязка к году и кооперативу.
2. Реализовать провайдеры КУДИР для разделов 3–7, либо один провайдер на группу секций — решить при проектировании DTO.
3. Подключить вывод PDF и Excel для новых секций в том же духе, что для разделов 1–2.
4. При необходимости — импорт и экспорт табличных данных. Продуктовые правила решать отдельно, не в этом файле.
