## 1. Словарик: английское название → русское → что это

Чтобы было понятно, о чём речь.

| EN | RU | Коротко |
|----|-----|--------|
| **Cooperative** | Садоводческое товарищество (СТ) | Одна организация — одно СТ. Tenant в системе. |
| **LandPlot** | Земельный участок | Участок принадлежит одному СТ (cooperative_id). |
| **Owner** | Владелец | Физ. лицо или юр. лицо; к СТ не привязан, только через участок. |
| **PlotOwnership** | Право/доля собственности на участок | Кто какой долей владел участком и в какой период (valid_from, valid_to). |
| **Meter** | Счётчик (прибор учёта) | Электричество, вода, газ и т.д. Привязан к владельцу. |
| **Payment** | Платёж | Факт оплаты по участку: сумма, дата, статус (создан/подтверждён/отменён). |
| **Accrual** | Начисление | Начисленная сумма по участку за период (взнос, услуга и т.д.). |
| **Balance** | Баланс / задолженность | Не таблица. Считаем: начисления минус платежи по участку. |
| **MeterReading** | Показание счётчика | Одно значение: счётчик, дата, показание. |
| **ContributionType** | Вид взноса | Справочник: членский, целевой и т.д. Нужен, чтобы тип начисления хранить. |

**Сейчас не делаем (добавим позже):**  
**Audit / OperationLog** (аудит, журнал операций) — кто что когда менял.  
**Report / Reporting** (отчёт, отчётность) — печатные отчёты, экспорт, сохранённые отчёты.

## 2. Что добавить в модель данных (Phase 2)

Сейчас в ядре уже есть: **Cooperative**, **LandPlot**, **Owner**, **PlotOwnership**, **Meter**. Этого достаточно для участков и владельцев.

Чтобы делать **платежи**, **начисления** и **показания счётчиков**, нужно добавить сущности:

| Сущность (EN) | По-русски | Что хранить в ней |
|---------------|-----------|-------------------|
| **Payment** | Платёж | Участок (land_plot_id), сумма, дата оплаты, статус; при желании — кто платил (owner_id). |
| **Accrual** | Начисление | Участок (land_plot_id), сумма, период (или даты), тип взноса (contribution_type_id), статус. |
| **MeterReading** | Показание счётчика | Счётчик (meter_id), значение, дата снятия показания. |
| **ContributionType** | Вид взноса | Справочник: название, код (членский, целевой и т.д.). |

**Balance (баланс)** — отдельной таблицы не делаем. Считается так: по участку сумма начислений минус сумма платежей. Если потом понадобятся «снимки на дату» — тогда отдельно обсудим.

**Аудит (OperationLog)** и **отчёты (Report)** — не добавляем в эту фазу, сделаем позже.

---

## 3. Связи (целостность)

- **Привязка к СТ (tenant):** по-прежнему только у **LandPlot** (cooperative_id). Платёж и начисление привязываем к **участку** (land_plot_id) — так мы знаем, к какому СТ они относятся (через участок → СТ).
- **Payment** — обязательно участок (land_plot_id); кто плательщик — по желанию (owner_id или поле payer).
- **Accrual** — участок (land_plot_id) и вид взноса (contribution_type_id, когда введём ContributionType).
- **MeterReading** — только счётчик (meter_id).

## 4. Модули и что в них делать

Кратко: границы модулей и первые шаги (API, правила, тесты). **Без аудита и отчётов.**

### 4.1. Land Management (управление участками и владельцами)

**Сущности:** Cooperative, LandPlot, PlotOwnership, Owner.

**Шаги:** API для CRUD по СТ, участкам, владельцам и правам собственности (доли, периоды). Правила: номер участка в рамках СТ один; доли по участку на дату в сумме не больше 1. Тесты: участок без владельца, назначение владельца, смена владельца.

---

### 4.2. Payments (платежи)

**Сущности:** Payment, LandPlot, Owner (плательщик — по желанию).

**Шаги:** API: создать/подтвердить/отменить платёж; список по участку и периоду. Правила: сумма > 0; статусы created → confirmed → cancelled. Тесты: создание оплаты и пересчёт баланса, отмена, частичная оплата.

---

### 4.3. Accruals (начисления)

**Сущности:** Accrual, LandPlot, ContributionType.

**Шаги:** API: создать начисления по участкам (одним или пачкой), список по участку/периоду, корректировка. Правила: сумма, период, статус. Тесты: одно начисление, массовое начисление, влияние на баланс.

---

### 4.4. Meters (счётчики и показания)

**Сущности:** Meter, MeterReading, Owner.

**Шаги:** API: CRUD счётчиков (привязка к владельцу), ввод показаний, история по счётчику. Правила: например, показания не убывают. Тесты: ввод показаний, проверка на «обратное» показание.

---

### 4.5. Reporting (отчётность)

**Сейчас не делаем.** Добавим позже: отчёты по периоду, должники, экспорт и т.д.

---

### 4.6. Administration (настройки СТ)

**Сущности:** Cooperative (настройки), пользователи/роли (вне ядра модели).

**Шаги:** API для настроек СТ (название и т.д.); разграничение доступа (администратор / председатель / казначей). **Журнал операций (аудит) не делаем в этой фазе.**

---

## 5. Порядок разработки (простыми словами)

1. **Дописать модель данных** — добавить в схему сущности: Payment, Accrual, MeterReading, ContributionType. Обновить диаграмму и schema-viewer. Аудит и отчёты не добавляем.

2. **Сделать участки и владельцы** (Land Management) — чтобы можно было выбирать участок при платеже и начислении.

3. **Сделать начисления** (Accruals) — тогда появится «долг» по участку.

4. **Сделать платежи** (Payments) — тогда можно гасить долг и считать баланс (начисления − платежи).

5. **Сделать счётчики и показания** (Meters) — ввод показаний, история. Связь с начислениями (по тарифу) — по необходимости позже.

6. **Отчётность и аудит** — не в этой фазе, добавим позже.

Итого: сначала модель Phase 2 (без аудита и отчётов), потом Land Management → Accruals → Payments → Meters. Остальное — потом.
