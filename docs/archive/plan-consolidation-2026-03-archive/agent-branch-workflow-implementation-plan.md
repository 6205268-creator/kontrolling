# План внедрения: работа агентов в ветках и ревью перед слиянием

**Назначение документа:** Пошаговый план действий для внедрения варианта 1 (Git-ветка + ревью перед merge). Документ предназначен для проверки другой нейросетью или человеком: можно одобрить целиком, дополнить или скорректировать перед выполнением.

**Дата:** 2026-03-05  
**Контекст:** Задача Ledger-ready MVP и любые будущие изолированные задачи агентов должны выполняться в отдельной ветке; слияние в основную ветку — только после явного одобрения пользователя и прохождения ревью архитектуры.

---

## 1. Глоссарий (для анализа и проверки)

| Термин | Значение |
|--------|----------|
| **Ветка (branch)** | Отдельная линия истории изменений в Git. Позволяет разрабатывать задачу изолированно, не трогая «основной» код. |
| **main / master** | Основная ветка проекта. В неё попадает только проверенный код. Обычно одна из них — «главная» (в этом проекте в CI указаны обе). |
| **Коммит (commit)** | Сохранённая «снимок» изменений в репозитории с подписью и сообщением. |
| **Merge (слияние)** | Внесение изменений из одной ветки в другую (например, из `feature/ledger-ready-mvp` в `main`). После merge код из ветки задачи становится частью main. |
| **Ревью (review)** | Проверка изменений перед принятием: тесты, линтеры, чек-лист архитектуры. Выполняется человеком или агентом (architecture-guardian). |
| **Architecture Guardian** | Роль/правило в проекте: чек-лист проверки границ слоёв, финансового ядра, модели данных, тестов. Вызывается вручную (@architecture-guardian). |
| **Architecture Linter** | Скрипт `backend/app/scripts/architecture_linter.py`: проверяет критические нарушения (API не импортирует infrastructure, Accrual/Payment используют FinancialSubject, регистрация моделей, domain без фреймворков, LandPlot без прямого FK на Owner). Запуск из каталога backend: `python -m app.scripts.architecture_linter`. |

---

## 2. Цели плана

1. **Запрет агенту коммитить в main/master.** Агент, выполняющий изолированную задачу, работает только в указанной ветке. Коммит в main допустим только после явного одобрения пользователя (на практике merge делает пользователь или по его команде).
2. **Добавить проверку архитектуры перед «готово».** Помимо pytest и ruff, перед принятием изменений запускать architecture linter и (по этапам) ревью architecture-guardian.
3. **Зафиксировать процесс ревью по этапам.** После каждого этапа задачи: тесты и линтеры зелёные → вызов ревью по затронутым модулям → при ошибках передать замечания агенту для исправления → повторная проверка. Переход к следующему этапу или merge только после «всё зелёное».
4. **Применить один и тот же способ ко всем агентам.** Любая изолированная задача выполняется по этому workflow: ветка → работа только в ней → ревью после этапа/перед merge → merge только по явному одобрению.
