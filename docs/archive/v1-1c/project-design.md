# Дизайн проекта Controlling (v2)

Система учёта хозяйственной деятельности **садоводческих товариществ (СТ)** Республики Беларусь.  
Автоматизация: начисление взносов, учёт платежей, контроль задолженностей, управление участками, приборы учёта.

---

## 1. Цели системы (v1)

1. **Учёт начислений** — расчёт и хранение начислений по видам взносов и услуг.
2. **Контроль оплат и задолженностей** — регистрация платежей, анализ долгов в реальном времени.
3. **Управление участками и владельцами** — ведение актуального состояния участков с поддержкой множественных владельцев.
4. **Учёт приборов учёта** — хранение показаний и истории счётчиков.
5. **Учёт расходов товарищества** — ремонт, зарплаты, материалы.
6. **Финансовая аналитика и отчётность** — должники, движение средств.
7. **Мультитенантность** — несколько СТ в одной установке.

## 2. Архитектура

```
┌─────────────────────────────────────────────────────────┐
│                    Пользователь (браузер)                │
└─────────────────────┬───────────────────────────────────┘
                      │ HTTPS
┌─────────────────────▼───────────────────────────────────┐
│              Frontend (Vue 3 SPA)                        │
│  Nginx: раздаёт статику + проксирует /odata/ на 1С       │
└─────────────────────┬───────────────────────────────────┘
                      │ OData v3 (HTTP/HTTPS)
┌─────────────────────▼───────────────────────────────────┐
│           1С:Enterprise (Backend + БД)                   │
│  Конфигурация + Веб-сервис OData + Разделитель данных    │
│  База данных: файловая или сервер 1С                     │
└─────────────────────────────────────────────────────────┘
```

**Принцип разделения ответственности:**
- **1С** — единственный источник правды: данные, бизнес-логика, аудит, хранение.
- **Frontend** — только отображение и пользовательский ввод, работает через OData.
- **Никакого промежуточного backend'а** — OData из коробки покрывает потребности.

## 4. 1С: конфигурация и OData

### 4.1. Конфигурация

За основу берётся **1С:Управление нашей фирмой (УНФ) для Беларуси** с расширениями.

Объекты, которые ведутся в 1С:

| Объект 1С | Соответствует | Комментарий |
|-----------|---------------|-------------|
| Справочник **Организации** | Cooperative (СТ) | Разделитель данных |
| Справочник **Контрагенты** | Owner (Владелец) | Физ./юр. лицо |
| Справочник **УчастокСТ** | LandPlot | Доп. справочник (расширение) |
| Регистр сведений **ВладениеУчастком** | PlotOwnership | Периодический, дробные доли |
| Справочник **Счётчики** | Meter | Доп. справочник |
| Регистр сведений **ПоказанияСчётчиков** | MeterReading | Периодический |
| Справочник **ВидыВзносов** | ContributionType | Членский, целевой и т.д. |
| Документ **НачислениеВзносов** | Accrual | Движения по рег. накопления |
| Документ **ПоступлениеОплаты** | Payment | Приходный документ |
| Документ **Расход** | Expense | Расходный документ |
| Регистр накопления **ВзаиморасчётыСТ** | FinancialSubject balance | Остатки = задолженность |

### 4.2. Публикация OData

1С публикует OData-сервис штатным механизмом (`/odata/standard.odata/`).

Для работы frontend'а нужно:
- Включить публикацию через веб-сервер Apache/IIS
- Добавить нужные объекты в список публикуемых (Administration → OData)
- Настроить CORS-заголовки на веб-сервере (или через nginx)
- Включить HTTP-аутентификацию 1С (форм-аутентификация или Basic)

### 4.3. Мультитенантность

Используется стандартный механизм **разделителя данных** (общие реквизиты с разделением).  
Каждое СТ = отдельная зона данных в одной базе. Пользователь при входе получает доступ только к своему СТ.

## 5. Frontend

### 5.1. Что делает

- Красивый SPA поверх 1С OData
- Отображает данные из 1С в понятном пользователю виде
- Позволяет создавать/редактировать объекты через OData-запросы (POST/PATCH)
- Работает в браузере, без установки клиента 1С

### 5.2. Структура проекта

```
frontend/
├── src/
│   ├── api/            # OData клиент и запросы к 1С
│   ├── stores/         # Pinia (auth, cooperatives, plots, ...)
│   ├── views/          # Страницы
│   ├── components/     # Переиспользуемые компоненты
│   ├── types/          # TypeScript-типы (соответствуют объектам 1С)
│   └── router/         # Vue Router
├── package.json
├── tsconfig.json
└── vite.config.ts
```

### 5.3. OData-клиент

Отдельный модуль `src/api/odata.ts` — обёртка над axios:
- Формирование OData URL (`$filter`, `$select`, `$expand`, `$orderby`, `$top`, `$skip`)
- Обработка ошибок 1С (OData возвращает ошибки в своём формате)
- Типизированные методы для каждой сущности

### 5.4. Аутентификация

Используется **форм-аутентификация 1С** через OData:
1. Frontend отправляет `POST /a/st/?sso` с логином/паролем
2. 1С возвращает сессионный cookie
3. Все последующие запросы идут с cookie
4. Logout — очистка сессии на стороне 1С

Пользователи заведены в 1С, роли — через права доступа 1С.

## 6. Ролевая модель

Роли определяются в 1С (профили доступа):

| Роль | Доступ | Ограничения |
|------|--------|-------------|
| **Казначей** | Начисления, платежи, расходы своего СТ | Только своё СТ |
| **Председатель** | Просмотр всех данных своего СТ, отчёты | Не создаёт финансовые документы |
| **Администратор** | Все СТ, настройки | Системный уровень |

Frontend скрывает/показывает кнопки и разделы на основе роли, полученной при входе.

## 7. Модель данных

Концептуальная модель данных описана в `docs/data-model/`:
- `entities-minimal.md` — минимальный набор сущностей
- `schema-viewer.html` — интерактивная схема (открыть в браузере)

Ключевые принципы модели остаются в силе:

| Принцип | Реализация |
|---------|------------|
| **FinancialSubject как ядро** | Все финансовые операции — через финансовый субъект |
| **PlotOwnership** | Множественные владельцы с долями и периодами |
| **Мультитенантность** | Разделитель данных 1С (Организация) |
| **Архивация вместо удаления** | Статусы: archived/cancelled, hard delete запрещён |
| **Историзация** | Журнал регистрации 1С + версионирование объектов |
