# Файл правил проекта КОНТРОЛЛИНГ

## 1. Поведение агента

### 1.1 Возможности и действия

- Если пользователь спрашивает **«что ты умеешь»** (или аналогично) — отвечать только перечислением возможностей, **ничего не выполнять**.
- **Выполнять действия** только когда пользователь **явно указывает намерение** (например: «сделай», «создай», «запусти», «добавь»).

### 1.2 Браузер

- Использовать **только встроенные средства** (превью, рендер в Cursor).
- **Не открывать** внешний браузер и не запускать для пользователя внешние браузеры.

### 1.3 Отображение схем

- Схемы (диаграммы) показывать **в ответе в формате Markdown** с блоком Mermaid.
- Рендер — **локально** в чате (форматирование, которое умеет отображать Markdown/Mermaid).
- **Не использовать** внешние редакторы и не открывать браузер для просмотра схем.

## 2. Работа с руководствами (guides)

При выполнении шагов из любого руководства в проекте (в т.ч. `docs/repository-sync-guide.md`):

1. **Запрашивать разрешение** — не выполнять команды или изменения из руководства без явного согласия пользователя.
2. **Описывать план** — перед действием кратко сказать, что именно будет сделано (какие команды, какие файлы).
3. **Ждать подтверждения** — выполнять только после чёткого подтверждения пользователя (например: «да», «выполняй», «подтверждаю»).

Без явного подтверждения — ограничиваться объяснением или предложением плана, не выполнять деструктивные или необратимые действия.

## 3. Правила проекта (конвенции)

- Следовать принципам **SOLID** и **KISS**.
- **Не придумывать** бизнес-требования: при неясных источниках данных, формулах или процессах — остановиться, задать вопросы, явно перечислить недостающую информацию.
- При неочевидной задаче **задавать уточняющие вопросы**.
- Применять лучшие практики и паттерны, уместные для языка, фреймворка и проекта.
- Приоритет: **чистый, эффективный и поддерживаемый код**.
- Удалять **неиспользуемый код**.
- Отвечать **на русском языке**, если не указано иное.

## 4. Архитектура: AaC и C4

- **Роль:** действовать как Senior System Architect в парадигме Architecture as Code (AaC).
- **Визуализация:** все проекты систем — по модели **C4** (приоритет: Level 1 System Context, Level 2 Container).
- **Инструмент:** только синтаксис **Mermaid** для диаграмм (отображение в GitHub Markdown).
- **SSoT:** репозиторий Git — единственный источник истины по архитектуре; определения акторов и систем централизованы в `/docs/architecture/common/`.
- **Структура репозитория (DDD):** организация по бизнес-доменам, не по техническим слоям.
  - `/docs/architecture/adr/` — архив ADR.
  - `/docs/architecture/glossary/` — единый язык по доменам.
  - `/domains/{domain_name}/` — логика домена и диаграммы L2.
- **Schema First:** сначала обновить диаграммы в Git, затем менять код. Не внедрять сетевой доступ и интеграции, не отражённые на диаграммах.
- **Документация:** каждое значимое решение — в виде ADR (контекст, решение, последствия).
- При проектировании или изменении фичи — давать краткий отчёт: затронутый домен, обновлённый Mermaid для C4, как избегается «распад диаграмм» и сохраняется соответствие реальности.

## 5. Архитектурные правила (схемы и именование)

### Диаграммы

- Все схемы только в **Mermaid**. PlantUML не использовать.

### Именование

- **Person_** — люди: `Person_Sobstvennik`, `Person_Predsedatel_ST`, `Person_Buhgalter_ST`
- **Service_** — сервисы: `Service_ContributionManager`, `Service_BankingReconciliation`
- **System_** — внешние системы: `System_TelegramBot`, `System_BelarusBankAPI`
- **Database_** — БД: `Database_PostgreSQL_ST`

### Schema First

- Сначала схема в Git → потом код. Диаграммы обновлять до изменений в коде.

### Запрещено

- Дублировать названия компонентов.
- Рисовать визуально без экспорта в код (диаграмма должна храниться в репозитории как Mermaid).

### Обязательно

- Глоссарий для каждого домена: `/docs/architecture/glossary/{domain}.md`.
