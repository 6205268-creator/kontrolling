# Дизайн проекта Controlling

Система учёта хозяйственной деятельности **садоводческих товариществ (СТ)** — кооперативы, законодательство **Беларусь**. В проекте используется только аббревиатура **СТ** (Садоводческое Товарищество). Учёт: участки, собственники, начисления, оплаты, приборы учёта, пени, платёжные документы, отчётность. Исходное описание и блоки — в `docs/source-material/БИЗНЕС_ЛОГИКА_И_СТРУКТУРА_БД.md`; концептуальная модель — также в `docs/source-material/model-data.txt`. При переносе в проект терминологию СНТ из исходников заменяем на СТ. Работа по модели ведётся и в `docs/project-design.md`, и в `docs/data-model/`.

## 1. Принятые решения (базовый стек)

### 1.1. Бэкенд
- **Язык:** Python  
- **Фреймворк:** Django  
- **Назначение:** REST API для фронта, бизнес-логика, работа с БД.

### 1.2. База данных
- **СУБД:** PostgreSQL.

### 1.3. Фронтенд
- **Стек:** React (SPA).  
- **Next.js** на старте не используем; при необходимости добавим позже (SSR, SEO).

### 1.4. Архитектура
- Фронт и бэкенд **разделены**: отдельное React-приложение и Django как API.
- Взаимодействие по **REST API** (Django — единственный backend на первом этапе).
- Идея «всё отдельно» заложена в расчёт на возможное развитие в сторону более модульной/микросервисной архитектуры позже.
- **Примечание:** разделы «Технологический стек», «Микросервисы», «CQRS» и т.п. из исходного материала (`docs/source-material/`) не применяются — принят стек из п. 1 (Django, PostgreSQL, React).

## 3. Авторизация (роли входа)

- **Роли** учитываются только для входа в систему и доступа к функциям (кто что может делать). Должности в СТ (председатель, бухгалтер и т.д.) в модели данных пока не трогаем.
- **Роли входа:** председатель, казначей, администратор. Вход по логину и паролю.
- **Председатель** и **казначей** — пользовательские роли с правами на работу в рамках своего СТ.
- **Администратор** — права председателя и казначея плюс расширенные возможности (настройки, просмотр). Администратор может видеть данные по всем товариществам (с фильтрами); казначей видит только своё СТ (реализацию разграничения добавим в процессе).
- **2FA (двухфакторная аутентификация):** не делаем на старте; зарезервировать на будущее.

---

## 4. Масштаб (мультитенантность)

- Система рассчитана на **несколько товариществ (СТ)** в одной установке.  
- **Обязательное требование к модели данных:** принадлежность к СТ (tenant) задаётся только у **LandPlot** (cooperative_id). К СТ от владельца идём через участок (LandPlot → Cooperative). Связи «владелец → СТ» нет.

## 5. Дизайн модели данных (подход)

- Модель данных проектируем в том же подходе, что и архитектура приложения: **Architecture as Code (AaC)**.  
- **Визуальное представление связей между данными обязательно:** диаграммы должны давать наглядную картину того, как сущности связаны между собой.  
- **Инструмент:** Mermaid (диаграммы в репозитории, рендер в Markdown/GitHub).  
- **Schema First:** схема/диаграмма модели данных обновляется до изменений в коде БД и приложения.  
- Промпт для разработки концептуальной модели: **`docs/data-model/conceptual-model-prompt.md`** — использовать в дальнейшей работе по модели данных.
- Минимальный набор сущностей (ядро): **`docs/data-model/entities-minimal.md`** (английские имена, русские комментарии).
- Схема для просмотра в браузере: **`docs/data-model/schema-viewer.html`** (открыть файл в браузере).

### 5.1. Уточнения по модели (ядро)

- **Нормальная форма:** cooperative_id только у **LandPlot**. Связи «владелец → СТ» нет — к товариществу идём через участок.
- **Владелец (Owner)** вместо «физическое лицо (Person)»: владелец участка/счётчика может быть **физ. лицом**, **юр. лицом** (напр. исполком) или участок **без владельца** (owner_id пустой). Тип задаётся в **owner_type** (physical, legal, …). Владелец с СТ напрямую не связан.
- **Участок (LandPlot)** — принадлежит одному СТ (cooperative_id); привязан к **владельцу (Owner)** (owner_id) или без владельца (owner_id пустой). Один владелец может иметь несколько участков.
- **Счётчик (Meter)** — привязан к **владельцу (Owner)** (owner_id).
- **Член СТ (Member)**, **Линия (Line)**, **вид взноса (ContributionType)** в ядре не используются — при необходимости добавим позже. **Право собственности (PlotOwnership)** входит в ядро (связь участок ↔ владелец по периодам и долям).
- **Валюта:** только Беларусь — все суммы в **белорусских рублях (BYN)**. Мультивалютность не предусмотрена.

## 7. Цели системы (System Goals)

Высокоуровневые цели описывают, какую ценность система предоставляет СТ.

1. **Учёт начислений по участкам** — расчёт и хранение начислений по видам взносов и услуг.
2. **Контроль оплат и задолженностей** — регистрация платежей и анализ долгов.
3. **Управление участками и владельцами** — ведение актуального состояния участков.
4. **Учёт приборов учёта** — хранение показаний и истории счётчиков.
5. **Финансовая аналитика и отчётность** — формирование управленческих отчётов.
6. **История операций и аудит** — прослеживаемость изменений.
7. **Администрирование СТ** — управление настройками товариществ.

Каждая цель соответствует будущему крупному функциональному модулю.

## 10. Основные рабочие процессы (Workflows)

### 10.1. Регистрация оплаты

1. Казначей выбирает участок.
2. Вводит сумму и параметры платежа.
3. Система создаёт запись Payment.
4. Пересчитывается баланс участка.
5. Запись фиксируется в истории.

### 10.2. Начисление взноса

1. Казначей выбирает тип начисления.
2. Система формирует Accrual для выбранных участков.
3. Обновляются балансы.
4. Операция фиксируется в журнале.

### 10.3. Снятие показаний счётчика

1. Пользователь вводит показания.
2. Создаётся запись MeterReading.
3. При необходимости формируется начисление.

### 10.4. Формирование отчёта

1. Пользователь выбирает период.
2. Система агрегирует данные.
3. Генерируется отчёт.

## 12. Доменные события (Domain Events)

* **PaymentRegistered** — зарегистрирован платёж
* **PaymentCancelled** — платёж отменён
* **AccrualCreated** — создано начисление
* **BalanceUpdated** — обновлён баланс
* **OwnerChanged** — изменён владелец
* **MeterReadingRecorded** — записаны показания
* **ReportGenerated** — сформирован отчёт

События используются как концептуальная основа для логики и аудита.
