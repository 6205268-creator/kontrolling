# Дизайн проекта Controlling

Система учёта хозяйственной деятельности **садоводческих товариществ (СТ)** — кооперативы, законодательство **Беларусь**. Корректная аббревиатура: **СТ** (не СНТ). Учёт: участки, собственники, начисления, оплаты, приборы учёта, пени, платёжные документы, отчётность. Исходное описание и блоки — в `docs/source-material/БИЗНЕС_ЛОГИКА_И_СТРУКТУРА_БД.md` (терминологию СНТ при переносе в проект заменяем на СТ).

---

## 1. Принятые решения (базовый стек)

### 1.1. Бэкенд
- **Язык:** Python  
- **Фреймворк:** Django  
- **Назначение:** REST API для фронта, бизнес-логика, работа с БД.

### 1.2. База данных
- **СУБД:** PostgreSQL.

### 1.3. Фронтенд
- **Стек:** React (SPA).  
- **Next.js** на старте не используем; при необходимости добавим позже (SSR, SEO).

### 1.4. Архитектура
- Фронт и бэкенд **разделены**: отдельное React-приложение и Django как API.
- Взаимодействие по **REST API** (Django — единственный backend на первом этапе).
- Идея «всё отдельно» заложена в расчёт на возможное развитие в сторону более модульной/микросервисной архитектуры позже.

- **Роли:** администратор, казначей, председатель. Вход по логину и паролю.  
- **2FA (двухфакторная аутентификация):** не делаем на старте; зарезервировать на будущее.

---

## 4. Масштаб (мультитенантность)

- Система рассчитана на **несколько товариществ (СТ)** в одной установке.  
- **Обязательное требование к модели данных:** во всех основных таблицах должен быть ключ (поле), разделяющий данные по товариществу (tenant). Так данные каждого кооператива изолированы. При дальнейшей проработке модели — учитывать это с самого начала.

---

## 5. Дизайн модели данных (подход)

- Модель данных проектируем в том же подходе, что и архитектура приложения: **Architecture as Code (AaC)**.  
- **Визуальное представление связей между данными обязательно:** диаграммы должны давать наглядную картину того, как сущности связаны между собой.  
- **Инструмент:** Mermaid (диаграммы в репозитории, рендер в Markdown/GitHub).  
- **Schema First:** схема/диаграмма модели данных обновляется до изменений в коде БД и приложения.  
- Промпт для разработки концептуальной модели: **`docs/data-model/conceptual-model-prompt.md`** — использовать в дальнейшей работе по модели данных.
- Минимальный набор сущностей (ядро): **`docs/data-model/entities-minimal.md`** (английские имена, русские комментарии).
- Схема для просмотра в браузере: **`docs/data-model/schema-viewer.html`** (открыть файл в браузере).

### 5.1. Уточнения по модели (ядро)

- **Физическое лицо (Person)** не общее для нескольких СТ: каждое лицо принадлежит **одному** садоводческому товариществу. СТ — полностью автономные единицы; физические лица — часть структуры кооператива, а не отдельная общая справочная сущность.
- **Банки, контрагенты, госорганы (налоги и т.д.)** в перспективе могут быть общей подсистемой для всех СТ; это не касается сущности «физическое лицо».
- **Физическое лицо** и **Член СТ (Member)** — две разные сущности; связь между ними **1:1**: одно физическое лицо не может быть дважды членом одного и того же товарищества (один человек = один член в данном СТ).
- **Один и тот же человек** может владеть **несколькими участками** в рамках одного СТ (связь через право собственности на участок).
