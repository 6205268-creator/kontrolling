# Техническое задание: матрица «жизнь СТ» + раннер сценария (демо, регрессия, обучение)

**Версия документа:** 0.2  
**Дата:** 2026-03-20  
**Статус:** черновик; артефакты Фазы A созданы в репозитории (см. §5.1, §13)  
**Проект:** Controlling (учёт в садовых товариществах, суммы в BYN)

---

## 1. Назначение и цели

### 1.1 Зачем это нужно

1. **Единый «снимок правды»** о том, какие жизненные ситуации товарищество должно уметь отражать в системе, и **на какой стадии** это отражено в продукте (реализовано / частично / нет / на пересмотр).
2. **Опора для развития продукта:** явные «дыры» по строкам матрицы → кандидаты в бэклог, без смешения с кодом.
3. **Будущий автоматизированный прогон:** сценарий года (календарный) через **реальные пользовательские пути** (преимущественно **HTTP**, как у пользователя) с **гибридом** (рутина через API, критичные для обучения места — через UI в браузере).
4. **Будущее интерактивное обучение:** тот же текст событий и шагов может стать основой подсказок для пользователя; на первом этапе достаточно заложить **раздельный канал «субтитров»** (не встраивать в UI Controlling).

### 1.2 Что явно не является целью v0 документа

- Не фиксировать **бизнес-правила** (порядок погашения взносов, виды начислений и т.д.), если они не утверждены владельцем продукта и в ADR/ТЗ модуля. Исполнитель **не выдумывает** правила.
- Не описывать конкретные **эндпоинты FastAPI** и поля DTO до аудита кода: в ТЗ указаны **типы шагов** и **места уточнения в репозитории**.

## 2. Термины

| Термин | Смысл |
|--------|--------|
| **Матрица жизни** | Один основной файл (Markdown-таблица или CSV): строка = событие жизни СТ + статус внедрения в системе. |
| **EVT-ID** | Стабильный идентификатор строки матрицы, например `EVT-014`. |
| **Сценарий года** | Логическое покрытие календарного года (январь–декабрь) последовательностью событий; может быть шире, чем то, что уже автоматизировано. |
| **Раннер** | Программа/набор скриптов, читающая машиночитаемое описание шагов (на втором этапе) и выполняющая действия против развёрнутого backend/frontend. |
| **Профиль прогона** | Режим запуска одного и того же сценария с разной глубиной и скоростью (`validate`, `smoke`, `full`, и т.д.). |
| **Дыра (gap)** | Событие или шаг, для которого в продукте **нет** адекватной сущности/экрана/API; прогон не обязан падать — шаг помечается и попадает в отчёт. |
| **Окно субтитров** | Отдельное от приложения Controlling средство отображения текста шага для наблюдателя (локальная страница / мини-сервер). |

## 3. Заинтересованные стороны и зоны ответственности

| Роль | Ответственность в матрице |
|------|---------------------------|
| **Владелец / автор сценария** | События жизни, даты в сценарии, формулировки «что должно быть в системе», статус «на пересмотр» с точки зрения жизни СТ. |
| **Казначей (эксперт предметной области)** | Дополнения и уточнения жизненных веток; новые строки с новыми EVT-ID. |
| **Разработчик** | Колонки реализации: реализовано/частично/нет, обходные пути, ссылки на задачи, привязка к коду; реализация раннера и логов. |

**Правило редактирования:** новые жизненные кейсы от казначея оформляются **новой строкой** с **новым EVT-ID**; старые ID не переиспользовать под другое значение.

## 4. Ограничения и допущения сценария (согласовано в обсуждении)

1. **Горизонт времени:** календарный год **январь–декабрь** (условный год жизни СТ).
2. **Масштаб:** ориентир **~10 участков**, имена и фамилии вымышленные, но последовательны в сценарии.
3. **Владение участком:** **один владелец на участок** (без долевой/совместной собственности в сценарии v0).
4. **Распределение платежей по статьям:** в ранней версии **не автоматизировать** умное распределение; предполагается **ручная работа казначея** в UI там, где продукт это позволяет. В матрице это отражается отдельными строками/чекпоинтами.
5. **Деньги:** в домене проекта — **BYN**; в тексте сценария избегать путаницы с «рублями».
6. **Источник правды по событиям:** **один файл-матрица** (без дублирования «второго сценария»), чтобы не запутаться. Расширение — новыми строками и версией в шапке.

## 5. Артефакт 1: файл матрицы (обязательный к созданию в репозитории)

### 5.1 Расположение (рекомендация)

- Каталог: `docs/scenario/` — **создан**
- Матрица: **`life-year-matrix.md`** (24 события EVT-001…024, кварталы, персонажи)
- Промпт для новой сессии: **`PROMPT-new-session.md`**
- Вложения: **`docs/scenario/files/`** (пример `bank-2024-05-05-sample.csv`)
- Индекс папки: **`README.md`**

### 5.2 Обязательные колонки

| Колонка | Содержание |
|---------|------------|
| **EVT-ID** | Уникальный стабильный код (`EVT-001`, …). |
| **Дата (в сценарии)** | Условная дата в году (может быть интервал, если нужно: «март, 1–15»). |
| **Событие жизни** | Простым языком: что произошло в СТ. |
| **Ожидаемый результат в системе** | Что казначей должен уметь увидеть/сделать в Controlling после события. |
| **Статус реализации** | Одно из согласованных значений (см. §5.3). |
| **Как сейчас обходим (если частично)** | Конкретика: какой экран, ручной ввод, временный процесс. |
| **Заметки / дыры** | Чего не хватает в модели или UI; формулировка для бэклога. |
| **Связь** | Опционально: issue, ADR, модуль (`land_management`, `payments`, …). |

Опциональные колонки:

- **Утверждение сценария** — например: `черновик` | `утверждено для прогона` | `идея от казначея` (чтобы раннер не трогал черновики).
- **Профили** — какие профили раннера должны включать строку (`smoke`, `full`), когда появится машиночитаемая привязка.

### 5.3 Статусы реализации (словарь)

Рекомендуемый набор значений в ячейке **«Статус реализации»**:

- `реализовано` — есть однозначный путь в UI и/или API, соответствующий смыслу строки.
- `частично` — смысл события закрывается обходным путём или урезанной моделью; описать в «Как сейчас обходим».
- `не реализовано` — в продукте нет способа отразить событие без фикции.
- `на пересмотр` — неясны правила или формулировка жизненного события; возможно, после казначея или разработчика статус сменится.

Исполнитель **не меняет** статус без согласования с владельцем/экспертом, кроме случаев технической верификации («в коде нет маршрута X») — тогда в «Заметках» фиксируется причина.

### 5.4 Версионирование матрицы

В **шапке файла** (первые строки до таблицы):

- Версия матрицы (например `0.2`).
- Дата последнего существенного изменения.
- Краткий **changelog** (3–5 bullet) или отсылка к коммиту.

- В матрице в колонке «Связь» или в «Заметках» — имя файла.

## 7. Артефакт 3: раннер и профили прогона (вторая фаза после матрицы)

### 7.1 Принципы

1. **Как пользователь:** критические для обучения действия выполняются через **браузер** (Playwright или аналог), с теми же итоговыми эффектами в БД, что при ручной работе.
2. **Рутина:** массовые операции (например, создание 10 участков) **не** обязаны выполняться десятками кликов; допускается **вызов HTTP API** (или подготовительный сид), затем в субтитрах инструкция «открой список участков и проверь».
3. **Субтитры:** отдельное окно/страница **не внутри** Vue-приложения Controlling, чтобы не усложнять продуктовый UI.
4. **Интерактивный режим:** паузы и кнопка «Дальше», настраиваемые задержки, опциональная подсветка элемента перед кликом (технически — на стороне Playwright).
5. **Повторяемость:** каждый полный прогон предполагает **известное начальное состояние БД** (чистая БД или согласованный сброс). Детали — в инструкции проекта (`AGENTS.md`, pytest, `seed_db`).

### 7.2 Профили (один сценарий — разные режимы)

| Профиль | Назначение |
|---------|------------|
| `validate` | Проверка целостности машиночитаемого сценария (когда появится): схема, уникальность `step_id`, существование файлов вложений. Без изменения БД. |
| `smoke` | Короткая цепочка для частых прогонов (PR / локально). |
| `full` | Полный календарный сценарий (все исполняемые шаги). |
| `api` | Максимум через HTTP без браузера; UI-шаги пропуск или замена, где возможно. |
| `interactive` | Демо для человека: субтитры + «Дальше» + увеличенные задержки. |
| `gaps` | Акцент на отчёте по строкам/шагам с типом «дыра»; не требовать 100% зелёного для целей планирования. |

Реализация профилей — через теги на шагах или отдельный список `step_id` для `smoke`.

### 7.3 Идемпотентность и идентификаторы

- У каждого **шага автоматизации** (когда появится YAML/JSON рядом с матрицей): стабильный `step_id`, связь с **EVT-ID** матрицы (один к одному или один EVT → несколько шагов).
- Повторный прогон на **чистой** БД даёт тот же результат; повторный прогон на **непустой** БД вне скоупа v0 — описать политику отдельно (обычно запрещено или только `smoke` на изолированной БД).

### 7.4 Логирование и отчёт (обязательные свойства)

Единый **`run_id`** на прогон.

**Файл `summary.json` (или эквивалент):**

- `run_id`, время начала/конца, профиль, версия матрицы, версия раннера, commit hash приложения (если доступен).
- Счётчики: успех / провал / пропуск / gap.
- Список упавших `step_id` и EVT-ID.

**Журнал шагов (например `steps.jsonl`):** одна строка JSON на шаг:

- `step_id`, `evt_id`, timestamp, тип (`api` | `ui` | `checkpoint`),
- статус,
- для API: метод, path, код ответа, краткая выжимка (без секретов),
- для UI: краткое описание действия, путь к скриншоту,
- при ошибке: сообщение, путь к trace/screenshot.

**Скриншоты:** каталог `artifacts/runs/<run_id>/`.

**Классификация провала:**

- Ошибка продукта / регрессия.
- Ошибка сценария (неверный селектор, устаревший шаг).
- Ожидаемая **дыра** — не считать провалом всего прогона, если политика `gaps`; в `full` политику согласовать (часто: дыры = жёлтый итог, падение только на обязательных шагах).

## 8. Связь матрицы и раннера

**Фаза A (текущий приоритет):** только матрица в `docs/scenario/` — без автоматизации.

**Фаза B:** к строкам EVT с статусом `реализовано` или осмысленным `частично` добавляется машиночитаемое описание шагов (отдельный файл или секция), **не дублируя** прозу: в матрице остаётся человекочитаемое событие, в YAML — технические вызовы.

**Правило:** если строка `не реализовано` или `на пересмотр`, автоматический шаг либо отсутствует, либо помечен `gap` и не ломает `smoke`.

## 9. Git-воркфлоу (рекомендация для владельца)

- Отдельная **ветка** для работы над матрицей и будущим раннером, например `docs/scenario-life-year` или `scenario/matrix-and-runner`.
- Изменения вливать в основную ветку разработки через **Pull Request**, чтобы история правок таблицы была видна на GitHub.
- Редактирование на GitHub возможно для мелких правок; крупные таблицы удобнее вести локально.

Раннер (когда появится) логично держать в отдельном подкаталоге репозитория, например `e2e-scenario/` или `tools/scenario-runner/`, с собственным `package.json` / `pyproject` — **не смешивать** с зависимостями backend без необходимости.

## 12. Открытые вопросы (уточнить у владельца продукта / по коду)

1. Точный перечень **уже существующих** сущностей и экранов для: банковской выписки, ручного распределения платежа, начислений «по решению собрания с участка», счётчиков и тарифа — зафиксировать в матрице после аудита.
