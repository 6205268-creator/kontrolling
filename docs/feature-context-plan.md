# План: дерево фич и минимальный контекст для экономии токенов

**Цель:** выполнять задачи по отдельным фичам с **минимально достаточным контекстом**, не подгружая весь проект. Это снижает расход токенов и ускоряет ответы.

**Идея:** для каждой фичи (или группы фич) задать явный **пакет контекста** — список файлов и правил, которые нужно прочитать перед выполнением задачи по этой фиче. Дерево фич позволяет ориентироваться и выбирать нужный пакет.

## 2. Пакет контекста (что подгружать для фичи)

Для каждой фичи задаётся:

| Элемент | Описание |
|--------|----------|
| **Обязательные файлы** | Файлы, без которых нельзя корректно выполнить фичу (фрагмент плана реализации, связанные модели, схемы, тесты). |
| **Опциональные файлы** | Дизайн, модель данных — только если нужны уточнения по бизнес-логике. |
| **Правила** | Только релевантные `.cursor/rules` (например, project-conventions — всегда; архитектура — только для фич по моделям/API). |
| **Зависимости** | Номера фич, от которых эта зависит (например, API LandPlots зависит от моделей LandPlot, PlotOwnership, Cooperative, Owner). |

**Принцип:** в контекст попадает **только** описание самой фичи (из плана реализации), перечисленные в пакете файлы кода/схем и минимальный набор правил. Полный `project-design.md` и полный `entities-minimal.md` — не по умолчанию, а только если фича явно в пакете или запрос требует бизнес-уточнений.

## 3. Где хранить пакеты контекста

Рекомендуемая структура:

```
docs/
  context-tree/
    README.md              # Оглавление: дерево фич + как пользоваться
    manifest.json         # (опционально) Машиночитаемый список фича → файлы
    features/
      F01.md              # Фича 1 — контекст
      F02.md
      ...
      F35.md
```

**Формат одного `features/Fxx.md`:**

- Заголовок: номер и название фичи.
- Блок **Контекст:** список путей к файлам (от корня репозитория).
- Блок **Правила:** какие правила из `.cursor/rules` применять.
- Блок **Зависимости:** номера фич (для справки или для каскадной подгрузки).
- При необходимости — краткая выжимка по сущностям/эндпоинтам (1–2 абзаца), чтобы не тянуть целый `project-design.md`.

Так агент (или ты вручную) по номеру фичи открывает один маленький файл и по нему подгружает только перечисленные артефакты.

## 4. Как использовать (workflow)

1. **Задача привязана к фиче**  
   Пример: «Сделай фичу 13» или «Доработай API участков (фича 13)».  
   → Открываешь `docs/context-tree/features/F13.md` и подгружаешь в контекст только перечисленные там файлы и правила. Выполняешь шаги из соответствующего фрагмента `project-implementation.md` (желательно этот фрагмент тоже включить в пакет или вынести в F13.md).

2. **Задача без указания фичи**  
   Сначала по описанию задачи определяешь фичу (или блок дерева), затем берёшь пакет контекста для этой фичи. При необходимости подгружаешь пакет зависимостей (например, только модели, без всего API).

3. **Сокращение объёма плана в контексте**  
   Вместо целого `project-implementation.md` в контекст можно подставлять только **секцию одной фичи** (например, «Фича 13: API endpoints — LandPlots»). Эту секцию можно дублировать в `features/F13.md` или генерировать из одного источника (если позже вынесешь фичи в отдельные файлы).

---

## 5. Шаги внедрения (план работ)

| Шаг | Действие | Результат |
|-----|----------|------------|
| 1 | Зафиксировать дерево фич в `docs/context-tree/README.md` | Оглавление и навигация по блокам |
| 2 | Для каждой фичи 1…35 создать `docs/context-tree/features/Fxx.md` с полями: контекст (файлы), правила, зависимости | Пакеты контекста готовы к использованию |
| 3 | Решить, хранить ли полный текст шагов фичи в Fxx.md или только ссылку на `project-implementation.md` + номера строк | Меньший дублирование vs быстрый доступ без открытия большого файла |
| 4 | Добавить правило или инструкцию в `.cursor/rules` или в README: «при задаче по фиче N загружать контекст из `docs/context-tree/features/FN.md`» | Единый способ использования |
| 5 | (Опционально) Собрать `manifest.json`: id фичи → массив путей файлов; при необходимости подгружать контекст по манифесту | Автоматизация для инструментов |

## 6. Пример пакета контекста (Фича 13)

**Фича 13: API endpoints — LandPlots (CRUD с PlotOwnership)**

**Контекст (файлы):**

- `docs/project-implementation.md` — только секция «Фича 13» (или копия в F13.md)
- `docs/data-model/entities-minimal.md` — фрагменты: LandPlot, PlotOwnership, Cooperative, Owner (или целиком, если файл небольшой)
- `backend/app/models/land_plot.py`
- `backend/app/models/plot_ownership.py`
- `backend/app/models/cooperative.py`
- `backend/app/models/owner.py`
- `backend/app/schemas/` — схемы для LandPlot, PlotOwnership (если уже есть)
- `backend/app/api/` — роутеры/эндпоинты для land_plots (если уже есть)
- `backend/tests/` — тесты, относящиеся к LandPlots API

**Правила:**  
`project-conventions.mdc`, при изменении модели — `architecture-rules.mdc` (Schema First).

**Зависимости:** Фичи 2, 3 (модели Cooperative, Owner, LandPlot, PlotOwnership); при наличии отдельной фичи по схемам — она тоже.

В результате для «сделай фичу 13» не нужно подгружать весь `project-design.md`, все 35 фич и весь backend — только перечисленное выше.

---

## 7. Ожидаемый эффект

- **Меньше токенов на запрос:** контекст ограничен одной фичей и её зависимостями.
- **Стабильное качество:** у фичи есть явный чек-лист (шаги из плана) и все нужные артефакты в одном пакете.
- **Масштабируемость:** новые фичи добавляются как новые узлы дерева и новые файлы в `features/`.

После подтверждения плана можно перейти к шагу 1 (README и дерево в `context-tree`) и затем к массовому заполнению `features/F01.md` … `F35.md` по шаблону выше.
