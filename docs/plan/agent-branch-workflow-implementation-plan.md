# План внедрения: работа агентов в ветках и ревью перед слиянием

**Назначение документа:** Пошаговый план действий для внедрения варианта 1 (Git-ветка + ревью перед merge). Документ предназначен для проверки другой нейросетью или человеком: можно одобрить целиком, дополнить или скорректировать перед выполнением.

**Дата:** 2026-03-05  
**Контекст:** Задача Ledger-ready MVP и любые будущие изолированные задачи агентов должны выполняться в отдельной ветке; слияние в основную ветку — только после явного одобрения пользователя и прохождения ревью архитектуры.

---

## 1. Глоссарий (для анализа и проверки)

| Термин | Значение |
|--------|----------|
| **Ветка (branch)** | Отдельная линия истории изменений в Git. Позволяет разрабатывать задачу изолированно, не трогая «основной» код. |
| **main / master** | Основная ветка проекта. В неё попадает только проверенный код. Обычно одна из них — «главная» (в этом проекте в CI указаны обе). |
| **Коммит (commit)** | Сохранённая «снимок» изменений в репозитории с подписью и сообщением. |
| **Merge (слияние)** | Внесение изменений из одной ветки в другую (например, из `feature/ledger-ready-mvp` в `main`). После merge код из ветки задачи становится частью main. |
| **Ревью (review)** | Проверка изменений перед принятием: тесты, линтеры, чек-лист архитектуры. Выполняется человеком или агентом (architecture-guardian). |
| **Architecture Guardian** | Роль/правило в проекте: чек-лист проверки границ слоёв, финансового ядра, модели данных, тестов. Вызывается вручную (@architecture-guardian). |
| **Architecture Linter** | Скрипт `backend/app/scripts/architecture_linter.py`: проверяет критические нарушения (API не импортирует infrastructure, Accrual/Payment используют FinancialSubject, регистрация моделей, domain без фреймворков, LandPlot без прямого FK на Owner). Запуск из каталога backend: `python -m app.scripts.architecture_linter`. |

---

## 2. Цели плана

1. **Запрет агенту коммитить в main/master.** Агент, выполняющий изолированную задачу, работает только в указанной ветке. Коммит в main допустим только после явного одобрения пользователя (на практике merge делает пользователь или по его команде).
2. **Добавить проверку архитектуры перед «готово».** Помимо pytest и ruff, перед принятием изменений запускать architecture linter и (по этапам) ревью architecture-guardian.
3. **Зафиксировать процесс ревью по этапам.** После каждого этапа задачи: тесты и линтеры зелёные → вызов ревью по затронутым модулям → при ошибках передать замечания агенту для исправления → повторная проверка. Переход к следующему этапу или merge только после «всё зелёное».
4. **Применить один и тот же способ ко всем агентам.** Любая изолированная задача выполняется по этому workflow: ветка → работа только в ней → ревью после этапа/перед merge → merge только по явному одобрению.

## 3. Действия по шагам

### Шаг 1. Создать правило Cursor для агентов при работе в ветке

**Где:** Новый файл в `.cursor/rules/`, например `agent-branch-workflow.mdc`.

**Содержание правила (суть):**

- **Когда применяется:** Когда пользователь или задание явно указывают «работай в ветке X» или «isolated task branch: X» (или аналог). То есть правило не запрещает обычную работу в main, а включается только при явно указанной рабочей ветке.
- **Обязательные требования к агенту:**
  - Работать **только** в указанной ветке (создавать её от main/master, если ещё нет; переключаться на неё перед правками).
  - **Не коммитить** в ветки `main` и `master`. Все коммиты делать только в текущую рабочую ветку.
  - **Не выполнять** merge в main/master самостоятельно. Слияние в main допускается только после явного указания пользователя («мержи», «одобряю слияние», «merge in main» и т.п.).
- **Уточнение:** Если в задании указано имя ветки (например, `feature/ledger-ready-mvp`), агент должен перед началом работы убедиться, что активная ветка — именно она (при необходимости создать её от актуального main и переключиться).

**Формат файла:** Cursor rule (YAML frontmatter: description, alwaysApply по желанию; основная часть — текст на русском/английском с чёткими пунктами).

**Результат:** Любой агент, получивший задание с указанием рабочей ветки, при загрузке правил из `.cursor/rules/` будет видеть запрет коммитов в main и запрет самостоятельного merge.

---

### Шаг 2. Добавить Architecture Linter в чек-лист «перед готово»

**Где:** Документ `docs/tasks/workflow-orchestration.md`, раздел 1.7 «Pre-Commit Checklist».

**Действие:** Добавить один пункт в список чек-листа:

- Запустить из каталога `backend`: `python -m app.scripts.architecture_linter`. Проверка должна пройти без ошибок (exit code 0). При ошибках — исправить нарушения перед тем, как считать задачу готовой к ревью/merge.

**Место в списке:** Логично после пунктов про pytest и ruff (проверки кода), до или рядом с пунктом про ревью project-orchestrator/архитектуру.

**Результат:** В едином источнике правды по workflow явно зафиксировано: перед «готово» и перед merge нужно проходить не только тесты и ruff, но и architecture linter.

---

### Шаг 3. Описать правило ревью после этапа в current-focus (и при необходимости в спецификации)

**Где:**  
- `docs/plan/current-focus.md` — секция про Ledger-ready MVP и «что делать завтра» / процесс.  
- При желании — краткая отсылка в `docs/tasks/IMPLEMENTATION_SPEC_LEDGER_READY.md` в блок «Передача агенту».

**Содержание:**

- После каждого этапа (1–5) выполнять по порядку:
  1. Агент запускает pytest и ruff (как уже предписано спецификацией).
  2. Пользователь (или скрипт) запускает `python -m app.scripts.architecture_linter` из каталога backend.
  3. Пользователь вызывает ревью по затронутым модулям, например:  
     «@architecture-guardian Проверь изменения в модулях [список модулей этапа] на соответствие архитектурным стандартам».  
     Для Этапа 1 — accruals, payments, expenses; для последующих этапов — добавить financial_core и др. по спецификации.
  4. Если architecture-guardian выявил нарушения: передать агенту список замечаний; агент вносит правки в той же ветке, после чего повторить пункты 1–3. К следующему этапу не переходить, пока ревью не пройдено.
  5. После прохождения ревью — переходить к следующему этапу (или, после этапа 5, к решению о merge в main).

**Результат:** В current-focus и при необходимости в спецификации зафиксировано: ревью после каждого этапа по модулям; при ошибках — передача замечаний агенту и повторная проверка.

---

### Шаг 4. Зафиксировать общий workflow для любых изолированных задач агентов

**Где:** Один из вариантов:

- Раздел в `docs/plan/development-plan-and-integrity.md` (как часть «Как сохраняется целостность» или «Предложения по дальнейшей работе»),  
- либо отдельный короткий документ `docs/plan/agent-isolated-task-workflow.md` со ссылкой из development-index или development-plan-and-integrity.

**Содержание (кратко):**

- Для любой изолированной задачи, выполняемой агентом:
  1. Пользователь создаёт ветку (или указывает имя ветки) и в задании пишет: «работай только в ветке &lt;имя&gt;» (при этом действует правило из шага 1).
  2. Агент перед каждым этапом (если этапов несколько) выводит план этапа и ждёт «да»/«начинай».
  3. После этапа: pytest, ruff, architecture_linter; затем ревью architecture-guardian по затронутым модулям; при замечаниях — правки в той же ветке и повтор проверок.
  4. Merge в main/master — только по явному одобрению пользователя после успешного прохождения всех этапов и ревью.

**Результат:** Один и тот же способ работы описан в docs и применим к любому агенту и любой изолированной задаче.

---

### Шаг 5. В задании Ledger-ready явно указать ветку и отсылку к правилу

**Где:** `docs/plan/current-focus.md` (и при необходимости первый абзац «Передача агенту» в `docs/tasks/IMPLEMENTATION_SPEC_LEDGER_READY.md`).

**Действие:**

- Указать имя рабочей ветки, например: `feature/ledger-ready-mvp`.
- Написать, что агент должен следовать правилу работы в ветке (см. `.cursor/rules/agent-branch-workflow.mdc`): работа только в этой ветке, не коммитить в main/master, не мержить в main без явного одобрения.
- Оставить или добавить требование: перед каждым этапом выводить краткий план и дожидаться «да»/«начинай».

**Результат:** При запуске задачи Ledger-ready и открытом current-focus агент видит ветку и общие правила; правило из шага 1 даёт запрет на main/master.

## 4. Порядок выполнения шагов

Рекомендуемый порядок, чтобы не было циклических зависимостей:

1. **Шаг 1** — создать правило `.cursor/rules/agent-branch-workflow.mdc`.
2. **Шаг 2** — добавить пункт про architecture linter в `workflow-orchestration.md`.
3. **Шаг 4** — описать общий workflow для изолированных задач в docs (development-plan-and-integrity или agent-isolated-task-workflow.md).
4. **Шаг 3** — в current-focus и при необходимости в IMPLEMENTATION_SPEC прописать ревью после этапа и передачу замечаний агенту.
5. **Шаг 5** — в current-focus и при необходимости в спецификации указать ветку `feature/ledger-ready-mvp` и отсылку к правилу.

Перед началом работы агента по Ledger-ready пользователю нужно один раз создать ветку `feature/ledger-ready-mvp` от актуального main (или master) и переключиться на неё (или указать агенту: «создай ветку feature/ledger-ready-mvp от main и работай в ней»).

## 5. Критерии готовности (для проверки другой нейросетью)

- Правило в `.cursor/rules/` явно запрещает коммиты в main/master и самостоятельный merge при работе в указанной ветке.
- В Pre-Commit Checklist в workflow-orchestration.md есть пункт про запуск architecture linter: `python -m app.scripts.architecture_linter` из каталога backend.
- В docs зафиксировано: после каждого этапа — тесты, ruff, architecture linter, ревью guardian по модулям; при ошибках — замечания агенту, правки, повтор.
- Общий workflow для изолированных задач описан так, чтобы его можно было применять к разным агентам и задачам.
- В задании Ledger-ready указана ветка и отсылка к правилу работы в ветке.

## 6. Открытые решения (для ревьюера)

- **Имя файла правила:** `agent-branch-workflow.mdc` — можно заменить на другое понятное имя в `.cursor/rules/`.
- **Имя ветки Ledger-ready:** в плане использовано `feature/ledger-ready-mvp`; при необходимости можно заменить на другое (например, `feat/ledger-ready`).
- **Где хранить общий workflow (шаг 4):** в development-plan-and-integrity или в отдельном `agent-isolated-task-workflow.md` — на выбор ревьюера или владельца docs.

Документ можно передать другой нейросети с формулировкой: «Проверь план на полноту и согласованность; одобри или предложи дополнения/правки перед внедрением».
