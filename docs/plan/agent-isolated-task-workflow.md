# Workflow для изолированных задач агентов

**Назначение:** Универсальный процесс для любой изолированной задачи, выполняемой агентом в отдельной ветке.

**Дата:** 2026-03-05
**Владелец:** Lead Architect
**Применение:** Все агенты при выполнении изолированных задач (фичи, рефакторинг, исправления ошибок)

---

## 1. Начало работы

### 1.1. Получение задачи

Пользователь формулирует задачу и (опционально) указывает имя ветки:

```
Задача: [описание задачи]
Ветка: feature/имя-задачи
```

Если ветка не указана — агент предлагает имя на основе задачи (например, `feature/ledger-ready-mvp`).

### 1.2. Проверка ветки

Агент проверяет существование ветки:

**Если ветка не существует:**

```
⚠️ Ветка [имя_ветки] не существует.

Создать её от main? (да/нет)
```

**Не создавать ветку без подтверждения пользователя.**

**Если ветка существует:**

```bash
git status
git log -n 5 --oneline
git branch --show-current
```

Сообщить пользователю:

```
📋 Ветка [имя_ветки] существует.

Последние коммиты:
- abc1234 Название последнего коммита
- def5678 Предыдущий коммит
...

Текущее состояние: [clean / есть изменения]

Продолжаю работу в этой ветке? (да/нет)
```

### 1.3. Переключение на ветку

После подтверждения пользователя:

```bash
# Если ветка существует
git checkout <имя_ветки>

# Если ветка не существовала (создать новую)
git checkout -b <имя_ветки> main
```

---

## 2. Выполнение этапов

Если задача состоит из нескольких этапов (например, Ledger-ready MVP: Этапы 1–5):

### 2.1. Перед началом этапа

Агент выводит план этапа:

```
📋 Этап N: [название этапа]

План:
1. [действие 1]
2. [действие 2]
3. [действие 3]

Начинаю? (да/нет)
```

**Не начинать этап без подтверждения «да»/«начинай».**

### 2.2. Выполнение изменений

Агент вносит изменения в рабочей ветке:
- Создание/изменение файлов
- Коммиты с понятными сообщениями

### 2.3. Проверки после этапа

После завершения изменений агент запускает проверки **в указанном порядке**:

| № | Проверка | Команда | Критерий успеха |
|---|----------|---------|-----------------|
| 1 | Тесты | `cd backend && pytest` | Все тесты пройдены (exit code 0) |
| 2 | Lint | `cd backend && ruff check . && ruff format --check .` | Без ошибок (exit code 0) |
| 3 | Architecture Linter | `cd backend && python -m app.scripts.architecture_linter` | Все проверки пройдены (exit code 0) |
| 4 | Seed Database | `cd backend && python -m app.scripts.seed_db` | Без ошибок (exit code 0) |

**Если проверка не прошла:**

- Исправить ошибку
- Перезапустить проверку
- Продолжить только после успеха

### 2.4. Ревью архитектуры

После успешного прохождения всех проверок агент вызывает ревью:

```
📋 Этап N завершён. Все проверки пройдены.

Вызываю @architecture-guardian для ревью по модулям: [список модулей]
```

**Формат запроса:**

```
@architecture-guardian Проверь изменения в модулях [модуль1, модуль2, ...] на соответствие архитектурным стандартам.
```

### 2.5. Исправление замечаний ревью

**Если architecture-guardian вернул ✅:**

```
✅ Ревью пройдено. Все архитектурные стандарты соблюдены.
```

Переход к следующему этапу (или завершение задачи).

**Если architecture-guardian вернул ⚠️ (предупреждения):**

```
⚠️ Обнаружены предупреждения:

1. [описание предупреждения]
2. [описание предупреждения]

Исправляю самостоятельно.
```

Агент исправляет **мелкие ошибки** (стиль, имена, форматирование) сразу:
1. Вносит правки в той же ветке
2. Перезапускает проверки (п. 2.3)
3. Повторно вызывает ревью (п. 2.4)

**Если architecture-guardian вернул ❌ (ошибки):**

```
❌ Обнаружены серьёзные нарушения архитектуры:

1. [описание ошибки 1]
2. [описание ошибки 2]

Требуется подтверждение пользователя на исправление.

Исправляю? (да/нет)
```

После подтверждения «да»:
1. Агент вносит правки в той же ветке
2. Перезапускает проверки (п. 2.3)
3. Повторно вызывает ревью (п. 2.4)
4. Сообщает результат пользователю

**Не переходить к следующему этапу, пока ревью не пройдено (✅).**

---

## 3. Завершение задачи

### 3.1. После всех этапов

Когда все этапы завершены и ревью пройдено:

```
✅ Все этапы задачи завершены.

Пройдены проверки:
- ✅ Тесты: N тестов пройдено
- ✅ Lint: без ошибок
- ✅ Architecture Linter: все проверки пройдены
- ✅ Seed Database: без ошибок
- ✅ Architecture Guardian: ревью пройдено

Ветка: [имя_ветки]
Коммитов: N

Готово к слиянию в main.
```

### 3.2. Merge в main

**Агент не выполняет merge самостоятельно.**

Агент сообщает пользователю:

```
📋 Задача завершена. Все проверки пройдены.

Для слияния в main выполните:
    git checkout main
    git merge [имя_ветки]
    git push

Или дайте команду: «мержи в main»
```

**Если пользователь явно сказал «мержи в main», «одобряю слияние», «merge in main»:**

Агент выполняет:

```bash
git checkout main
git merge [имя_ветки] --no-ff -m "Merge branch [имя_ветки]: [название задачи]"
git push
```

Сообщает результат:

```
✅ Слияние в main выполнено.

Коммит: [хэш коммита]
Сообщение: [текст]
```

---

## 4. Исключения

Правило **не применяется**, если:

- Пользователь явно сказал: «работай в main», «коммить в main»
- Задача не является изолированной (настройка окружения, документация без кода)
- Пользователь отключил правило явной командой

## 6. Чек-лист для пользователя

Перед тем, как дать задачу агенту:

- [ ] Сформулировать задачу чётко (что сделать, критерии готовности)
- [ ] Указать имя ветки (или позволить агенту предложить)
- [ ] Если задача многоэтапная — указать этапы или ссылку на спецификацию

После завершения задачи агентом:

- [ ] Проверить, что все тесты пройдены
- [ ] Проверить, что architecture linter прошёл
- [ ] Проверить сообщение о ревью architecture-guardian
- [ ] Принять решение о merge в main
