# Промпт для агента: начало и конец сессии

**Назначение:** Вставляй этот блок в начало каждого нового чата с агентом (или дай агенту как первую инструкцию в сессии). Агент будет знать, что делать **до** начала работы и **после** её завершения.

---

## Промпт (скопируй и вставь агенту)

```
Перед началом сессии и по её завершении действуй по этому чек-листу.

**В НАЧАЛЕ СЕССИИ (перед любой задачей):**

1. Открой и прочитай единую точку входа проекта: `docs/development-index.md`.
2. Посмотри секцию «Топ-5: что делать прямо сейчас» (4.2) — возьми задачу оттуда, если пользователь не дал конкретную задачу.
3. Если работаешь над задачей из текущего фокуса (например Ledger-ready) — открой `docs/plan/current-focus.md`: на какой ветке работать, какие этапы, какие проверки гнать.
4. **Работай только в своей ветке задачи.** Ветка создаётся под эту задачу (имя — из current-focus или по указанию пользователя). Если ветки ещё нет: предложи имя (например `feature/краткое-название`), спроси «создать ветку?»; после «да» создай ветку от main и переключись на неё. Все коммиты — только в эту ветку. Правило: `.cursor/rules/git-branch-policy.mdc` (не коммитить в main/master, не мержить без одобрения пользователя).
5. Кратко подтверди мне: какую задачу берёшь, из какого источника (Топ-5 / current-focus / по моей просьбе), **в какой ветке работаешь** (имя ветки; если создал новую — напиши это).

**ВО ВРЕМЯ РАБОТЫ:**

- Не разрешать коммиты в main и merge без моей команды.
- После каждого этапа — проверки (pytest, ruff, architecture_linter, seed_db) + ревью @architecture-guardian.
- Серьёзные правки (архитектура, границы модулей) — только после моего «да»; мелкие — можно править сразу.

**ПЕРЕД MERGE В MAIN:**

- Убедиться, что все этапы и проверки пройдены, ревью есть.
- Merge выполнять только после моей явной команды («мержи в main», «одобряю слияние»).

**ПО ЗАВЕРШЕНИИ СЕССИИ (когда закончил работу или останавливаемся на сегодня):**

1. Если закрыта задача из Топ-5 или из `docs/tasks/` — предложи обновить:
   - `docs/development-index.md` (секция Топ-5, если приоритеты изменились);
   - `docs/history/PENDING_GAPS.md` и `docs/history/RESOLVED_GAPS.md` (если закрыта дыра);
   - `docs/project-implementation.md` (если выполнена фича — отметить шаги [x] и поставить ✅).
2. Обнови `docs/plan/current-focus.md`: на чём остановились, какой этап следующий, что делать в следующей сессии. Если не работали по current-focus — предложи обновление на основе сделанного.
3. Напомни про проверки перед «готово» (если что-то меняли в коде): pytest, ruff, architecture_linter, seed_db; при архитектурных изменениях — ревью @architecture-guardian.
4. Кратко подведи итог сессии: что сделано, что осталось, что первым делом делать в следующий раз.
```

## Короткая версия (если нужно в два предложения)

**В начале:** Открой `docs/development-index.md`, посмотри Топ-5 и при необходимости `docs/plan/current-focus.md`. Работай только в ветке задачи (создай её от main, если нет — предложи имя и спроси перед созданием); подтверди задачу и имя ветки.  
**Во время:** Не коммитить в main, не мержить без команды; после каждого этапа — проверки + ревью; серьёзные правки — только после «да». Перед merge — все проверки и ревью пройдены, затем явно сказать «мержи в main».  
**В конце:** Предложи обновить индекс, current-focus, PENDING_GAPS/RESOLVED_GAPS или project-implementation по сделанному; подведи итог и напомни про проверки (pytest, ruff, architecture_linter) перед «готово».
