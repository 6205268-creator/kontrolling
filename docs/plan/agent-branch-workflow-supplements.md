### Дополнение к Шагу 1 (Создание правила git-branch-policy)

**Файл:** `.cursor/rules/git-branch-policy.mdc`

**Содержание правила:**

```markdown
# Git Branch Policy для агентов

## Когда применяется

- Когда пользователь указывает: «работай в ветке X» или «isolated task branch: X»
- Когда задача является изолированной (не срочный хотфикс в main)

## Обязательные требования

1. **Не коммитить в main/master** — все коммиты только в рабочую ветку
2. **Не выполнять merge в main** — только после явного одобрения пользователя
3. **Если ветка не существует** — спросить: «Ветка [имя] не существует. Создать от main?»
4. **Если ветка существует** — проверить статус и последние коммиты, сообщить пользователю

## Команды для проверки

```bash
git status
git log -n 5 --oneline
git branch --show-current
```
```

```markdown
# Architecture Guardian

## Роль

Проверка изменений на соответствие архитектурным стандартам проекта.

## Когда вызывается

- После завершения каждого этапа задачи
- Перед merge в main/master
- По явному запросу пользователя: «@architecture-guardian проверь [модули]»

## Чек-лист проверки

1. **Границы слоёв** — API не импортирует infrastructure, domain не зависит от фреймворков
2. **Финансовое ядро** — Accrual/Payment используют FinancialSubject, нет прямых начислений на участки
3. **Модель данных** — новые модели зарегистрированы в `db/register_models.py`, миграции созданы
4. **Тесты** — тесты покрывают новую логику, все тесты зелёные
5. **Согласованность с диаграммами** — если менялась модель, обновлены диаграммы в docs/data-model/

## Формат ответа

- ✅ Всё чисто — по всем пунктам
- ⚠️ Предупреждения — мелкие нарушения (стиль, имена)
- ❌ Ошибки — серьёзные нарушения (границы, архитектура)

При ошибках — указать конкретные файлы и строки.
```

## Исправление ошибок после ревью

Если architecture-guardian выявил нарушения:

1. **Мелкие ошибки** (стиль, имена, форматирование):
   - Агент исправляет сразу в той же ветке
   - Перезапускает pytest + ruff + architecture_linter
   - Сообщает пользователю: «Исправил N замечаний, все проверки зелёные»

2. **Серьёзные ошибки** (границы слоёв, архитектура, FinancialSubject):
   - Агент показывает пользователю список нарушений
   - Спрашивает: «Обнаружены серьёзные нарушения. Исправляю?»
   - После подтверждения «да» — вносит правки
   - Перезапускает все проверки и сообщает результат

### Дополнение к Шагу 4 (Общий workflow)

**Файл:** `docs/plan/agent-isolated-task-workflow.md`

**Содержание:**

```markdown
# Workflow для изолированных задач агентов

## Назначение

Универсальный процесс для любой изолированной задачи, выполняемой агентом в отдельной ветке.

## Процесс

### 1. Начало работы

1. Пользователь указывает задачу и (опционально) имя ветки
2. Агент проверяет существование ветки:
   - Если нет — спрашивает: «Создать ветку [имя] от main?»
   - Если да — проверяет статус и последние коммиты, сообщает пользователю
3. Агент переключается на рабочую ветку

### 2. Выполнение этапов

Для каждого этапа:

1. Агент выводит план этапа и ждёт «да»/«начинай»
2. Выполняет изменения в ветке
3. Запускает проверки:
   - `pytest` (все тесты зелёные)
   - `ruff check .` и `ruff format --check .` (без ошибок)
   - `python -m app.scripts.architecture_linter` (exit code 0)
   - `python -m app.scripts.seed_db` (без ошибок)
4. Вызывает `@architecture-guardian` для ревью по затронутым модулям
5. Исправляет замечания (мелкие — сразу, серьёзные — после подтверждения)
6. Повторяет проверки после исправлений
7. Переходит к следующему этапу только после «всё зелёное»

### 3. Завершение и merge

1. После всех этапов агент сообщает: «Все этапы завершены, проверки пройдены»
2. Пользователь явно одобряет merge: «мержи в main», «одобряю слияние»
3. Merge выполняет пользователь (или агент по явной команде)

## Ссылки

- Правило веток: `.cursor/rules/git-branch-policy.mdc`
- Ревью архитектуры: `.cursor/rules/agents/architecture-guardian.mdc`
- Pre-Commit Checklist: `docs/tasks/workflow-orchestration.md` (раздел 1.7)
```
