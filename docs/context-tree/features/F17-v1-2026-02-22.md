# Фича 17: API endpoints — Expenses (расходы СТ)

---
## Выполнено

**Версия:** 1
**Дата:** 2026-02-22

**Простыми словами:**
Реализован учёт расходов товарищества: создание, подтверждение и отмена расходов по категориям. Расходы СТ (ремонт дорог, зарплата, материалы) учитываются отдельно от начислений и платежей — не привязаны к финансовым субъектам и не влияют на баланс участков.

**Роль в структуре проекта (технически):**
- **Слой:** API (endpoints) + Service Layer
- **Зависимости:** F06 (Expense, ExpenseCategory модели), F09 (схемы), F10 (аутентификация)
- **Интеграция:**
  - Использует модели `app.models.expense.Expense`, `app.models.expense_category.ExpenseCategory`
  - Использует схемы `app.schemas.expense.ExpenseCreate/ExpenseInDB/ExpenseCategoryInDB`
  - Сервис `app.services.expense_service` — CRUD + смена статусов + категории
  - Подключен в `app.main.py` как `/api/v1/expenses`
- **Что использует:** JWT-токены, разграничение по ролям (admin/treasurer)
- **Что от неё зависит:** Отчётность по расходам СТ, анализ финансовой деятельности

---

**Цель:** Учёт расходов, категории, подтверждение/отмена.

## Реализация

### Созданные файлы:

1. **`backend/app/schemas/expense.py`** — Pydantic схемы:
   - `ExpenseCategoryBase` — name, code, description
   - `ExpenseCategoryCreate` — для создания категории
   - `ExpenseCategoryInDB` — полная схема с id, created_at
   - `ExpenseBase` — cooperative_id, category_id, amount, expense_date, document_number, description
   - `ExpenseCreate` — для создания расхода
   - `ExpenseUpdate` — для обновления (category_id, amount, document_number, description)
   - `ExpenseInDB` — полная схема с id, status, timestamps

2. **`backend/app/services/expense_service.py`** — сервисный слой:
   - `create_expense()` — создание со статусом "created"
   - `get_expense()` — получение по ID
   - `confirm_expense()` — смена статуса на "confirmed" (только из "created")
   - `cancel_expense()` — смена статуса на "cancelled" (из любого кроме "cancelled")
   - `get_expenses_by_cooperative()` — список по СТ
   - `get_expense_categories()` — список категорий
   - `get_expense_category()` — получение категории по ID

3. **`backend/app/api/v1/expenses.py`** — API endpoints:
   - `GET /api/v1/expenses/categories` — список категорий
   - `GET /api/v1/expenses/` — список расходов (cooperative_id)
   - `POST /api/v1/expenses/` — создание расхода (treasurer, admin)
   - `POST /api/v1/expenses/{id}/confirm` — подтвердить (treasurer, admin)
   - `POST /api/v1/expenses/{id}/cancel` — отменить (treasurer, admin)

4. **`backend/tests/test_api/test_expenses.py`** — 11 тестов:
   - Получение категорий
   - Создание расхода
   - Подтверждение (created → confirmed)
   - Отмена (confirmed → cancelled)
   - Отмена уже отменённого (400)
   - Подтверждение уже подтверждённого (400)
   - Получение по СТ
   - 403 для чужого СТ
   - 422 при сумме <= 0
   - Treasurer без cooperative_id (auto)
   - 404 при несуществующей категории

## Разграничение прав

| Endpoint | Метод | Роли |
|----------|-------|------|
| /expenses/categories | GET | Все авторизованные |
| /expenses/ | GET | Все авторизованные (фильтр по СТ) |
| /expenses/ | POST | treasurer, admin |
| /expenses/{id}/confirm | POST | treasurer, admin |
| /expenses/{id}/cancel | POST | treasurer, admin |

## Статусы расходов

```
created → confirmed → cancelled
    ↓         ↓
    └─────────┘
```

- **created** — создан (требует подтверждения)
- **confirmed** — подтверждён (учитывается в отчётности)
- **cancelled** — отменён (не учитывается)

## Особенности

- Расходы **не привязаны** к FinancialSubject
- Расходы **не влияют** на баланс FinancialSubject (total_accruals/total_payments)
- Расходы — отдельный финансовый поток (учёт расходов СТ как юрлица)
