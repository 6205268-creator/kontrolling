# Фича 16: API endpoints — Payments (платежи)

---
## Выполнено

**Версия:** 1
**Дата:** 2026-02-22

**Простыми словами:**
Реализована регистрация платежей от владельцев в счёт погашения задолженности. Платежи создаются со статусом "confirmed" и сразу учитываются в балансе. Можно отменить платёж (статус "cancelled") и получить список платежей по участку, владельцу или всему СТ.

**Роль в структуре проекта (технически):**
- **Слой:** API (endpoints) + Service Layer
- **Зависимости:** F05 (Payment, Owner модели), F09 (схемы), F10 (аутентификация), F14 (FinancialSubject)
- **Интеграция:**
  - Использует модели `app.models.payment.Payment`, `app.models.owner.Owner`
  - Использует схемы `app.schemas.payment.PaymentCreate/PaymentInDB`
  - Сервис `app.services.payment_service` — CRUD + отмена + фильтрация
  - Подключен в `app.main.py` как `/api/v1/payments`
  - Использует `app.services.balance_service.get_financial_subject` для проверки доступа
- **Что использует:** JWT-токены, разграничение по ролям (admin/treasurer)
- **Что от неё зависит:** F14 (балансы — confirmed платежи учитываются в total_payments), отчётность по платежам

---

**Цель:** Регистрация и отмена платежей, список по субъекту/владельцу.

## Реализация

### Созданные файлы:

1. **`backend/app/schemas/payment.py`** — Pydantic схемы:
   - `PaymentBase` — financial_subject_id, payer_owner_id, amount, payment_date, document_number, description
   - `PaymentCreate` — для создания
   - `PaymentUpdate` — для обновления (document_number, description)
   - `PaymentInDB` — полная схема с id, status, timestamps

2. **`backend/app/services/payment_service.py`** — сервисный слой:
   - `register_payment()` — регистрация со статусом "confirmed"
   - `get_payment()` — получение по ID
   - `cancel_payment()` — смена статуса на "cancelled"
   - `get_payments_by_financial_subject()` — список по субъекту
   - `get_payments_by_owner()` — список по владельцу
   - `get_payments_by_cooperative()` — список по СТ (через join с FinancialSubject)

3. **`backend/app/api/v1/payments.py`** — API endpoints:
   - `GET /api/v1/payments/` — список (financial_subject_id, owner_id или cooperative_id)
   - `POST /api/v1/payments/` — регистрация платежа (treasurer, admin)
   - `POST /api/v1/payments/{id}/cancel` — отмена (treasurer, admin)

4. **`backend/tests/test_api/test_payments.py`** — 10 тестов:
   - Регистрация платежа
   - Отмена платежа
   - Отмена уже отменённого (400)
   - Получение по финансовому субъекту
   - Получение по владельцу
   - Получение по СТ
   - 403 для чужого СТ
   - 400 без параметров
   - 422 при сумме <= 0
   - Treasurer видит только своё СТ

### Изменённые файлы:

## Статусы платежей

```
confirmed → cancelled
```

- **confirmed** — подтверждён, учитывается в балансе (total_payments)
- **cancelled** — отменён, не учитывается в балансе

## Влияние на баланс

```
balance = total_accruals - total_payments

где total_payments = SUM(Payment.amount) WHERE status = 'confirmed'
```
