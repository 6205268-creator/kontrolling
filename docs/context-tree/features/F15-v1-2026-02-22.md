# Фича 15: API endpoints — Accruals (начисления)

---
## Выполнено

**Версия:** 1
**Дата:** 2026-02-22

**Простыми словами:**
Реализовано управление начислениями: создание, применение и отмена. Начисления создаются со статусом "created", затем применяются (статус "applied") и начинают учитываться в балансе. Можно отменить начисление или создать сразу несколько начислений для разных участков (массовое создание).

**Роль в структуре проекта (технически):**
- **Слой:** API (endpoints) + Service Layer
- **Зависимости:** F05 (Accrual, ContributionType модели), F09 (схемы), F10 (аутентификация), F14 (FinancialSubject)
- **Интеграция:**
  - Использует модели `app.models.accrual.Accrual`, `app.models.contribution_type.ContributionType`
  - Использует схемы `app.schemas.accrual.AccrualCreate/AccrualInDB/AccrualBatchCreate`
  - Сервис `app.services.accrual_service` — CRUD + смена статусов + массовое создание
  - Подключен в `app.main.py` как `/api/v1/accruals`
  - Использует `app.services.balance_service.get_financial_subject` для проверки доступа
- **Что использует:** JWT-токены, разграничение по ролям (admin/treasurer)
- **Что от неё зависит:** F14 (балансы — applied начисления учитываются в балансе), отчётность

## Реализация

### Созданные файлы:

1. **`backend/app/schemas/accrual.py`** — Pydantic схемы:
   - `AccrualBase` — financial_subject_id, contribution_type_id, amount, accrual_date, period_start, period_end
   - `AccrualCreate` — для создания
   - `AccrualUpdate` — для обновления (amount, period_end)
   - `AccrualInDB` — полная схема с id, status, timestamps
   - `AccrualBatchCreate` — для массового создания (список AccrualCreate)

2. **`backend/app/services/accrual_service.py`** — сервисный слой:
   - `create_accrual()` — создание со статусом "created"
   - `get_accrual()` — получение по ID
   - `apply_accrual()` — смена статуса на "applied" (только из "created")
   - `cancel_accrual()` — смена статуса на "cancelled" (из любого кроме "cancelled")
   - `get_accruals_by_financial_subject()` — список по субъекту
   - `get_accruals_by_cooperative()` — список по СТ (через join с FinancialSubject)
   - `mass_create_accruals()` — массовое создание в одной транзакции

3. **`backend/app/api/v1/accruals.py`** — API endpoints:
   - `GET /api/v1/accruals/` — список (financial_subject_id или cooperative_id)
   - `POST /api/v1/accruals/` — создание (treasurer, admin)
   - `POST /api/v1/accruals/batch` — массовое создание (treasurer, admin)
   - `POST /api/v1/accruals/{id}/apply` — применить (treasurer, admin)
   - `POST /api/v1/accruals/{id}/cancel` — отменить (treasurer, admin)

4. **`backend/tests/test_api/test_accruals.py`** — 10 тестов:
   - Создание начисления
   - Применение (created → applied)
   - Отмена (applied → cancelled)
   - Отмена уже отменённого (400)
   - Применение уже применённого (400)
   - Массовое создание
   - Получение по финансовому субъекту
   - Получение по СТ
   - 403 для чужого СТ
   - 400 без параметров

## Статусы начислений

```
created → applied → cancelled
    ↓         ↓
    └─────────┘
```

- **created** — создано, не учитывается в балансе
- **applied** — применено, учитывается в балансе (total_accruals)
- **cancelled** — отменено, не учитывается в балансе
