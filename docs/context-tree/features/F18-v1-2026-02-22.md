# Фича 18: API endpoints — Meters и показания

---
## Выполнено

**Версия:** 1
**Дата:** 2026-02-22

**Простыми словами:**
Реализовано управление счётчиками воды и электроэнергии: создание, просмотр, добавление показаний. При создании счётчика автоматически создаётся финансовый субъект для будущих начислений. Показания записываются с датой снятия, на одну дату возможно только одно показание.

**Роль в структуре проекта (технически):**
- **Слой:** API (endpoints) + Service Layer
- **Зависимости:** F04 (FinancialSubject), F07 (Meter, MeterReading модели), F09 (схемы), F10 (аутентификация)
- **Интеграция:**
  - Использует модели `app.models.meter.Meter`, `app.models.meter_reading.MeterReading`
  - Использует схемы `app.schemas.meter.MeterCreate/MeterInDB/MeterReadingCreate/MeterReadingInDB`
  - Сервис `app.services.meter_service` — CRUD счётчиков + показания + автосоздание FinancialSubject
  - Подключен в `app.main.py` как `/api/v1/meters`
  - FinancialSubject создаётся через `app.models.financial_subject.FinancialSubject`
- **Что использует:** JWT-токены, разграничение по ролям (admin/treasurer)
- **Что от неё зависит:** F19 (отчёты), начисления за коммунальные услуги

## Реализация

### Созданные файлы:

1. **`backend/app/schemas/meter.py`** — Pydantic схемы:
   - `MeterBase` — owner_id, meter_type, serial_number, installation_date, status
   - `MeterCreate` — для создания
   - `MeterUpdate` — для обновления
   - `MeterInDB` — полная схема с id, timestamps
   - `MeterReadingBase` — meter_id, reading_value, reading_date
   - `MeterReadingCreate` — для добавления показания
   - `MeterReadingInDB` — полная схема с id, created_at

2. **`backend/app/services/meter_service.py`** — сервисный слой:
   - `create_meter()` — создание счётчика + автосоздание FinancialSubject
   - `get_meter()` — получение по ID
   - `get_meters_by_owner()` — список по владельцу
   - `update_meter()` — обновление
   - `delete_meter()` — удаление
   - `add_meter_reading()` — добавление показания
   - `get_meter_readings()` — история показаний
   - `get_meter_with_financial_subject()` — счётчик с ФС

3. **`backend/app/api/v1/meters.py`** — API endpoints:
   - `POST /api/v1/meters/` — создать счётчик (treasurer, admin)
   - `GET /api/v1/meters/?owner_id={id}` — список по владельцу
   - `GET /api/v1/meters/{id}` — получение по ID
   - `POST /api/v1/meters/{id}/readings` — добавить показание
   - `GET /api/v1/meters/{id}/readings` — история показаний

4. **`backend/tests/test_api/test_meters.py`** — 11 тестов:
   - Создание счётчика воды (WATER)
   - Создание счётчика электроэнергии (ELECTRICITY)
   - Автосоздание FinancialSubject
   - Получение списка по владельцу
   - Получение по ID
   - 404 при несуществующем ID
   - Добавление показания
   - Получение истории показаний
   - 400 при дублировании даты
   - 400 при несуществующем владельце
   - 400 без owner_id

## Разграничение прав

| Endpoint | Метод | Роли |
|----------|-------|------|
| /meters/ | GET | Все авторизованные (требуется owner_id) |
| /meters/ | POST | treasurer, admin |
| /meters/{id} | GET | Все авторизованные |
| /meters/{id}/readings | GET | Все авторизованные |
| /meters/{id}/readings | POST | treasurer, admin |

## Типы счётчиков

- **WATER** — счётчик воды (FinancialSubject тип: WATER_METER)
- **ELECTRICITY** — счётчик электроэнергии (FinancialSubject тип: ELECTRICITY_METER)

## Статусы счётчиков

- **active** — активен/установлен
- **inactive** — не активен/снян

## Особенности

- FinancialSubject создаётся автоматически при создании счётчика
- FinancialSubject создаётся только если у владельца есть участок (PlotOwnership) или платежи
- Unique constraint на (meter_id, reading_date) — одно показание на дату
- reading_value >= 0 (проверка в схеме)
