# Лог архитектурных решений

Документ фиксирует принятые архитектурные решения с контекстом и последствиями. Детали — в ADR (когда они созданы).

## DEC-001: Модульная Clean Architecture вместо плоских app.models

- **Дата**: 2026-03-01 (фиксация по факту миграции)
- **Контекст**: Модели и репозитории перенесены из `app/models/` в модули `app/modules/{module}/infrastructure/models.py`. Документация и часть кода (seed, e2e) ещё ссылаются на app.models.
- **Рассмотренные варианты**: А) Оставить app.models как фасад над модулями. Б) Полностью перейти на модули, обновить все импорты и документацию. В) Два источника правды (не рекомендуется).
- **Принятое решение**: Каноническое размещение ORM-моделей — только в модулях (`infrastructure/models.py`). Регистрация для Alembic и тестов — через `db/register_models.py`. Скрипты наполнения и e2e должны быть приведены к импорту из модулей (или фасаду, если введён). Документация и правила — обновить под этот выбор.
- **Последствия**: Все новые модели добавляются только в модулях; в conftest импортируются модульные models; seed_db/seed_user и frontend e2e требуют правок (задача команде Backend).
- **Ссылки**: docs/history/PENDING_GAPS.md (источник правды для ORM), docs/tasks/TASK_Backend_20260301.md.

## DEC-002: Канонический технологический стек — FastAPI + Vue 3

- **Дата**: 2026-03-01 (фиксация по факту кодовой базы)
- **Контекст**: В docs/project-design.md указаны Django и React; в AGENTS.md и README — FastAPI и Vue 3. Реализация соответствует последним.
- **Рассмотренные варианты**: А) Считать каноничным project-design и переписать код. Б) Считать каноничным AGENTS.md/реализацию и привести project-design в соответствие.
- **Принятое решение**: Источник правды по стеку — AGENTS.md и фактическая кодовая база. docs/project-design.md должен быть обновлён (раздел технологического стека и развёртывания) до FastAPI + Vue 3 + PostgreSQL.
- **Последствия**: project-design.md перестаёт вводить в заблуждение новых разработчиков; обновление выполняет Architecture/Backend по заданию.
- **Ссылки**: docs/history/PENDING_GAPS.md, docs/tasks/TASK_Architecture_20260301.md.

## DEC-003: Единственный владелец глоссариев — Lead Architect

- **Дата**: 2026-03-01 (уточнение после TASK_Architecture)
- **Контекст**: Нужна «чистая» архитектура в смысле одного источника правды: кто принимает решения по артефактам (глоссарии, единый язык). Иначе проект разъезжается — глоссарии начнут менять кто попало и где попало; единый язык теряется.
- **Рассмотренные варианты**: А) Только Lead Architect. Б) Владелец фичи/модуля или архитектор. В) Совместно с ревью.
- **Принятое решение**: **Единственный владелец глоссариев — Lead Architect.** Только он создаёт и изменяет файлы в `docs/architecture/glossary/`. Никто другой (владелец модуля, разработчик, агенты) не вносит правки. Предложения по терминам передаются архитектору. Правило зафиксировано в architecture-rules.mdc, glossary/README.md и AGENTS.md с самого старта, чтобы предупредить размывание ответственности.
- **Последствия**: Все агенты и роли при необходимости изменить термин не правят глоссарии сами — передают предложение Lead Architect. Один человек — один источник решений по единому языку.
- **Ссылки**: docs/architecture/glossary/README.md, .cursor/rules/architecture/architecture-rules.mdc, AGENTS.md (Project Conventions), docs/tasks/TASK_Architecture_20260301.md.
