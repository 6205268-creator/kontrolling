## Детали по дырам

### Источник правды для технологического стека

- **Описание**: В `docs/project-design.md` указаны Django 5.x и React 18; в AGENTS.md и README — FastAPI и Vue 3. Новый разработчик не понимает, какой документ каноничен.
- **Почему ещё не устранена**: Не было явного владельца и политики актуализации дизайн-документа.
- **Что нужно для устранения**: Зафиксировать в ADR или в project-design.md, что канонический стек — FastAPI + Vue 3; обновить project-design.md (раздел «Технологический стек») в соответствии с AGENTS.md.
- **Блокеры**: Нет.

### Источник правды для размещения ORM-моделей

- **Описание**: Правила (backend-testing-and-pitfalls.mdc) и фичи (F12–F18 и др.) ссылаются на `app.models`; фактически модели в `modules/*/infrastructure/models.py`, регистрация через `db/register_models.py`. Скрипты `seed_db.py` и `seed_user.py` импортируют из `app.models` (папка удалена в миграции) — код в неконсистентном состоянии.
- **Почему ещё не устранена**: Миграция на Clean Architecture не доведена до документации и скриптов.
- **Что нужно для устранения**: Обновить правила и фичи на «модели в модулях»; привести seed_db/seed_user и e2e к импорту из модулей или единому фасаду; зафиксировать политику в ADR/AGENTS.md.
- **Блокеры**: Нет.

### Границы слоёв: зависимость API от Infrastructure

- **Описание**: `app/api/deps.py` импортирует `AppUserModel` из `app.modules.administration.infrastructure.models`. Presentation слой напрямую зависит от Infrastructure; направление зависимостей Clean Architecture нарушается.
- **Почему ещё не устранена**: Нет зафиксированного правила «тип текущего пользователя только из Domain/Application или фасад».
- **Что нужно для устранения**: Ввести интерфейс/тип в domain или application; API использовать только его; маппинг в Infrastructure. Зафиксировать в политике слоёв (ADR или docs/architecture).
- **Блокеры**: Нет.

### Роли окружений и политика переменных

- **Описание**: Нет документа «Environment Policy»: какие окружения (dev/stage/prod), какие переменные обязательны, кто владеет секретами, где хранятся .env для каждого окружения.
- **Почему ещё не устранена**: Не формализовано.
- **Что нужно для устранения**: Создать документ (например docs/architecture/environment-policy.md или раздел в DEPLOY.md) с матрицей окружений и переменных.
- **Блокеры**: Нет.

### Расположение .env и рабочий каталог

- **Описание**: `config.py` задаёт `env_file=".env"` (относительно текущей рабочей директории). При запуске из корня проекта `backend/.env` не подхватывается. Не зафиксировано: «запуск backend только из backend/» или использование пути относительно корня репозитория.
- **Почему ещё не устранена**: Не документировано.
- **Что нужно для устранения**: Явно указать в AGENTS.md/README: откуда читается .env и из какой директории нужно запускать backend; при необходимости изменить загрузку конфига (например путь к .env относительно файла конфига).
- **Блокеры**: Нет.

### Каноничность get_db и переопределение в тестах

- **Описание**: Сессию БД инжектируют через `app.api.deps.get_db` (реэкспорт из `app.db.session.get_db`). В тестах переопределяют оба — deps.get_db и db_session.get_db. Не объяснено, почему нужны оба переопределения и какой источник считается каноническим для роутов.
- **Почему ещё не устранена**: Нет явной политики «один вход для get_db» и «как тесты подменяют сессию».
- **Что нужно для устранения**: Задокументировать: каноническая точка — deps.get_db; в тестах переопределять только её, если все роуты берут сессию через deps; либо зафиксировать причину двух переопределений в backend-testing-and-pitfalls.mdc.
- **Блокеры**: Нет.

### ADR: индекс и процесс ведения

- **Описание**: В AGENTS.md и aac-c4-architecture указано «ADR для каждого значимого решения»; в репозитории только шаблон, нет ни одного принятого ADR и индекса. Неясно, кто ведёт ADR и по какому формату.
- **Почему ещё не устранена**: Не внедрён процесс.
- **Что нужно для устранения**: Создать docs/architecture/adr/README.md с индексом и правилами (когда создавать, формат, ответственный); добавить первый ADR (например, по стеку FastAPI+Vue или по модульной структуре).
- **Блокеры**: Нет.

### Глоссарии по доменам

- **Описание**: Правило «глоссарий для каждого домена»; есть только `docs/architecture/glossary/contributions.md`. Домены cooperative, land, financial, payments, expenses, meters, reporting, administration без глоссариев. Не закреплено, кто создаёт и обновляет глоссарии.
- **Почему ещё не устранена**: Не приоритизировано.
- **Что нужно для устранения**: Зафиксировать в архитектурном правиле: перечень доменов и ответственный за глоссарии; по возможности добавить заглушки или первые версии для ключевых доменов.
- **Блокеры**: Нет.

### Канонический seed и скрипты наполнения

- **Описание**: AGENTS.md называет единственным каноническим seed только `seed_db.py`; отдельно существует `seed_user.py`; docker-compose вызывает только seed_db. Не определено: один скрипт vs несколько, идемпотентность, кто поддерживает.
- **Почему ещё не устранена**: Не формализовано.
- **Что нужно для устранения**: Политика в AGENTS.md или ADR: один канонический entry point (например seed_db вызывает seed_user при необходимости) или явное разделение; требования к идемпотентности и поддержке.
- **Блокеры**: Нет.

### CI и переменные для тестов

- **Описание**: В `.github/workflows/backend-tests.yml` не задаётся DATABASE_URL; тесты полагаются на conftest.py, который выставляет SQLite до импорта app. Не документировано, что CI не требует PostgreSQL и что это ожидаемое поведение.
- **Почему ещё не устранена**: Не задокументировано.
- **Что нужно для устранения**: Кратко описать в AGENTS.md или в CI-комментарии: тесты backend используют in-memory SQLite через conftest; переменная DATABASE_URL в CI не обязательна.
- **Блокеры**: Нет.
